### Issue #1: Git import: reusing an existing clone fails when remote URL differs only by `.git` suffix

**Detail Bug ID**: `bug_f840e242-54ce-4170-9fa7-2e3e2efb4a73`
**File**: `backend/src/yinshi/services/git.py`
**Impact**: Medium | **Severity**: Minor | **Priority**: P2
**Status**: Fixed

#### What was the problem?

The clone reuse check compared remote URLs with a plain string comparison. That treated `https://host/repo` and `https://host/repo.git` as different repositories even though Git hosts treat them as the same target.

#### What was changed?

**Before:**
```python
if existing_remote.strip().rstrip("/") != url.strip().rstrip("/"):
    raise GitError("Destination already contains a clone of a different repository")
```

**After:**
```python
if not _remote_urls_match(existing_remote, url):
    raise GitError("Destination already contains a clone of a different repository")
```

#### How does the code work?

1. `_remote_urls_match()` normalizes both URLs.
2. The normalization strips whitespace, a trailing slash, and an optional `.git` suffix.
3. `clone_repo()` now reuses a valid clone when the two URLs are logically the same repository.

#### Why was this change needed?

Without normalization, users hit a false error during perfectly safe clone reuse. The fix makes `clone_repo()` consistent with the module's other remote URL comparison logic.

### Issue #2: Pi config upload can exhaust server memory by reading archives before enforcing the 50MB limit

**Detail Bug ID**: `bug_af40090e-6066-4728-b3e6-25beabdf2fed`
**File**: `backend/src/yinshi/api/settings.py`
**Impact**: High | **Severity**: Major | **Priority**: P1
**Status**: Fixed

#### What was the problem?

The upload endpoint called `await file.read()` before checking archive size. That let one request allocate the entire upload in memory, even when the upload should have been rejected.

#### What was changed?

**Before:**
```python
filename = file.filename or "pi-config.zip"
zip_data = await file.read()
return await import_from_upload(..., zip_data=zip_data, filename=filename)
```

**After:**
```python
filename = file.filename or "pi-config.zip"
zip_data = await _read_upload_bytes(file, max_bytes=_MAX_UPLOAD_BYTES)
return await import_from_upload(..., zip_data=zip_data, filename=filename)
```

#### How does the code work?

1. `_read_upload_bytes()` checks `file.size` when Starlette already knows the size.
2. It then reads the upload in 1 MB chunks.
3. If the running total exceeds the configured limit, it raises `HTTPException(413)` immediately.
4. The route closes the uploaded file in `finally` so temporary resources do not linger.

#### Why was this change needed?

The old code enforced the limit too late. The new code enforces the limit before building an oversized in-memory `bytes` object.

### Issue #3: OAuth concurrent multi-provider logins can intermittently fail with account_error due to users.email race

**Detail Bug ID**: `bug_6e10f596-f553-4aca-b091-026df838202c`
**File**: `backend/src/yinshi/services/accounts.py`
**Impact**: Medium | **Severity**: Major | **Priority**: P1
**Status**: Fixed

#### What was the problem?

Two concurrent OAuth callbacks for the same email could both see “no user row yet” and race on `INSERT INTO users (...)`. One callback succeeded. The other hit the `UNIQUE(users.email)` constraint and returned `account_error`.

#### What was changed?

**Before:**
```python
user_row = db.execute("SELECT id, email FROM users WHERE email = ?", (email,)).fetchone()
if user_row:
    ...
user_id = secrets.token_hex(16)
db.execute(
    "INSERT INTO users (id, email, display_name, avatar_url, encrypted_dek, last_login_at) "
    "VALUES (?, ?, ?, ?, ?, CURRENT_TIMESTAMP)",
    (user_id, email, display_name, avatar_url, encrypted_dek),
)
```

**After:**
```python
user_id, user_was_created = _create_user_record(
    db,
    email=email,
    display_name=display_name,
    avatar_url=avatar_url,
)
_insert_oauth_identity(
    db,
    user_id=user_id,
    provider=provider,
    provider_user_id=provider_user_id,
    email=email,
    provider_data_json=provider_data_json,
)
```

#### How does the code work?

1. `_create_user_record()` attempts the user insert once.
2. If SQLite raises `IntegrityError`, the code loads the already-created row by email and continues.
3. `_insert_oauth_identity()` inserts the provider identity, but tolerates the same identity being inserted concurrently by verifying that it belongs to the same user.
4. `_ensure_tenant_provisioned()` makes sure a newly discovered control-plane user also has a real per-user database on disk.

#### Why was this change needed?

The race did not lose data, but it produced spurious login failures. The new flow converges both concurrent requests on the same user record and keeps both provider identities.

### Issue #4: Git workspaces can be created empty when reusing an interrupted clone

**Detail Bug ID**: `bug_5a922937-e0b1-45f9-9162-4dfa3adc3943`
**File**: `backend/src/yinshi/services/git.py`
**Impact**: Medium | **Severity**: Major | **Priority**: P1
**Status**: Fixed

#### What was the problem?

A partially cloned repository could still pass `validate_local_repo()`. `clone_repo()` then swallowed a failed `git fetch --all` and returned the broken checkout. Later worktree creation could proceed from a repository with no remote refs and build an empty branch.

#### What was changed?

**Before:**
```python
try:
    with _git_askpass_env(access_token) as env:
        await _run_git(["fetch", "--all"], cwd=dest, env=env)
except GitError:
    pass
return dest
```

**After:**
```python
had_remote_refs_before_fetch = await _has_remote_refs(dest)
try:
    with _git_askpass_env(access_token) as env:
        await _run_git(["fetch", "--all"], cwd=dest, env=env)
except GitError as error:
    if not had_remote_refs_before_fetch:
        raise GitError("Existing clone is incomplete and could not be refreshed") from error
    return dest
if not had_remote_refs_before_fetch and not await _has_remote_refs(dest):
    raise GitError("Existing clone is incomplete and missing remote refs")
```

#### How does the code work?

1. `_has_remote_refs()` checks whether the checkout already has real fetched `refs/remotes/origin/*` refs.
2. If the checkout had no remote refs before refresh, `clone_repo()` now treats a failed fetch as a hard error.
3. Even after a successful fetch, the code rechecks for remote refs before accepting the clone.
4. Only a complete checkout is now eligible for reuse.

#### Why was this change needed?

A corrupted clone should fail loudly, not quietly poison the next workspace. The new guard stops empty worktrees at the reuse boundary.

### Issue #7: GitHub App install callback can be CSRF’d to link a victim’s installation to an attacker account

**Detail Bug ID**: `bug_1e4e25ba-4886-40d8-926e-9348ef83f9ab`
**File**: `backend/src/yinshi/api/auth_routes.py`
**Security vulnerability**: Yes
**Impact**: Critical | **Severity**: Critical | **Priority**: P0
**Status**: Fixed

#### What was the problem?

The callback trusted the signed `state` token alone. It never checked that the currently authenticated Yinshi user matched the user encoded inside `state`.

#### What was changed?

**Before:**
```python
user_id = _verify_github_install_state(state)
if not user_id:
    return RedirectResponse(url="/app?github_connect_error=invalid_state")
```

**After:**
```python
state_user_id = _verify_github_install_state(state)
if not state_user_id:
    return RedirectResponse(url="/app?github_connect_error=invalid_state")
current_user_id = _current_user_id(request)
if current_user_id != state_user_id:
    return RedirectResponse(url="/app?github_connect_error=invalid_state")
user_id = state_user_id
```

#### How does the code work?

1. The callback still validates the signed `state` token.
2. It now also reads the authenticated user from the `yinshi_session` cookie.
3. If those two user ids do not match, the callback rejects the request.
4. Only the authenticated initiator can bind a GitHub installation now.

#### Why was this change needed?

This was a real account-linking vulnerability. A signed state token proves integrity, but it does not prove that the browser making the callback is the same logged-in user who started the flow.

### Issue #8: Weak ENCRYPTION_PEPPER values are accepted, enabling offline brute-force of encrypted API keys

**Detail Bug ID**: `bug_17c9735b-d325-4cc8-81e5-f2e702446742`
**File**: `backend/src/yinshi/config.py`
**Security vulnerability**: Yes
**Impact**: High | **Severity**: Critical | **Priority**: P1
**Status**: Fixed

#### What was the problem?

The code validated that `ENCRYPTION_PEPPER` was hex, but it did not validate that the decoded value was long enough to be secure.

#### What was changed?

**Before:**
```python
if settings.encryption_pepper:
    try:
        bytes.fromhex(settings.encryption_pepper)
    except ValueError as exc:
        raise RuntimeError(...)
```

**After:**
```python
if settings.encryption_pepper:
    try:
        pepper_bytes = bytes.fromhex(settings.encryption_pepper)
    except ValueError as exc:
        raise RuntimeError(...)
    if len(pepper_bytes) < 32:
        raise RuntimeError(
            "ENCRYPTION_PEPPER must be at least 32 bytes (64 hex characters)"
        )
```

#### How does the code work?

1. Startup still rejects malformed hex.
2. Startup now also rejects decoded peppers shorter than 32 bytes.
3. `.env.example` now documents the setting so operators see the requirement in the default config file.

#### Why was this change needed?

A too-short pepper undermines every wrapped DEK in the database. The new startup check turns an unsafe deployment into an explicit boot failure.

### Issue #9: Auth: Logging out from one device revokes all sessions across devices

**Detail Bug ID**: `bug_1557ce71-089a-4019-9f8e-4e05a962fece`
**File**: `backend/src/yinshi/auth.py`, `backend/src/yinshi/api/auth_routes.py`
**Impact**: Medium | **Severity**: Minor | **Priority**: P2
**Status**: Fixed

#### What was the problem?

`/auth/logout` was supposed to revoke only the current session, but it called `revoke_auth_sessions(user_id)`, which revokes every active session for that user.

#### What was changed?

**Before:**
```python
user_id = _current_user_id(request)
if user_id is not None:
    revoke_auth_sessions(user_id)
```

**After:**
```python
session_identity = _current_session_identity(request)
if session_identity is not None:
    user_id, auth_session_id = session_identity
    revoke_auth_session(user_id, auth_session_id)
```

#### How does the code work?

1. `get_session_identity()` verifies the cookie and returns both `user_id` and `auth_session_id`.
2. `revoke_auth_session()` updates exactly one row in `auth_sessions`.
3. `/logout-all` still calls `revoke_auth_sessions()` and keeps the all-devices behavior.

#### Why was this change needed?

The old code violated the endpoint contract and logged users out on unrelated devices. The fix aligns the implementation with the route name and docstring.

### Issue #12: Frontend: Chunk load error recovery UI can blank-screen when sessionStorage is unavailable

**Detail Bug ID**: `bug_0082a526-944d-4d04-a3e2-6cea2333bd59`
**File**: `frontend/src/components/ChunkErrorBoundary.tsx`
**Impact**: Low | **Severity**: Minor | **Priority**: P3
**Status**: Fixed

#### What was the problem?

The error boundary called `shouldReloadForChunkError()` directly from `componentDidCatch()`. If `sessionStorage.setItem()` threw `QuotaExceededError` or `SecurityError`, the error boundary could throw while handling another error and fail to render recovery UI.

#### What was changed?

**Before:**
```tsx
if (
  shouldReloadForChunkError(
    window.sessionStorage,
    window.location.pathname,
    readEntryScriptSignature(),
  )
) {
  window.location.reload();
  return;
}
this.setState({ reloadRecommended: true });
```

**After:**
```tsx
let shouldReload = false;
try {
  shouldReload = shouldReloadForChunkError(
    window.sessionStorage,
    window.location.pathname,
    readEntryScriptSignature(),
  );
} catch (storageError: unknown) {
  if (!isStorageAccessError(storageError)) {
    throw storageError;
  }
  this.setState({ reloadRecommended: true });
  return;
}
```

#### How does the code work?

1. `isStorageAccessError()` recognizes the DOM exceptions that mean browser storage is unavailable.
2. `componentDidCatch()` now catches only that storage failure class.
3. When storage is unavailable, it falls back to the manual “Reload page” UI instead of trying to auto-reload.

#### Why was this change needed?

Error-boundary recovery code must not throw during recovery. This fix preserves the fallback UI in browsers with restricted session storage.

## Deferred Issues

### Issue #5: Sidecar warmup errors can corrupt the next query response stream

**Reason**: Cross-process protocol change. Needs coordinated Python + Node sidecar work and a compatibility plan for the warmup message contract.

**Recommendation**: Make warmup replies symmetric. Send an explicit success frame on the Node side and consume exactly one warmup response on the Python side.

### Issue #6: Container reaping can race with user requests, causing 500 errors and unsynchronized per-user containers

**Reason**: Shared-state locking fix touches container lifecycle paths and needs careful regression coverage around activity tracking and reaper timing.

**Recommendation**: Rework `destroy_container()` to acquire the same per-user lock used by `ensure_container()`, then add concurrency-focused tests around reaper overlap.

### Issue #10: Sidecar session creation errors can be misreported as user cancellations during stop-and-steer

**Reason**: Sidecar session lifecycle bug. Needs a focused JavaScript-side concurrency fix and regression test around stale `cancelRequested` state.

**Recommendation**: Delete stale session entries before awaiting replacement session creation, then add a targeted sidecar test for concurrent cancel + recreation failure.

### Issue #11: Agent UI can get stuck in “stopping” when cancelling a run if server-side abort hangs

**Reason**: Needs a design decision for how frontend state should recover when backend cancellation never confirms completion.

**Recommendation**: Add a client-side escape hatch that can close or replace the live stream locally after a timeout, then test retry behavior from the `stopping` state.
