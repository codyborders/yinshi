## Detailed Findings

### Bug 1: Prompt provider can resolve to null

**Plan reference**: `.context/attachments/plan.md`
**Impact**: High | **Severity**: Major | **Priority**: P0
**Status**: Fixed

#### What was the problem?

The prompt endpoint trusted the sidecar to always return a provider name. When the sidecar returned `None`, the backend passed that value into API-key lookup. Users then saw a confusing payment or key error instead of a clear validation failure.

#### What was changed?

**Before:**
```python
provider: str | None = resolved["provider"]
api_key, key_source = resolve_api_key_for_prompt(
    tenant.user_id, provider, platform_key,
)
```

**After:**
```python
provider: str | None = resolved["provider"]
if not provider:
    raise HTTPException(
        status_code=400,
        detail="Could not determine provider for model",
    )
api_key, key_source = resolve_api_key_for_prompt(
    tenant.user_id, provider, platform_key,
)
```

#### How does the code work?

1. The sidecar still resolves the requested model.
2. The API now checks whether the resolved provider is truthy.
3. If it is missing, the request stops with an HTTP 400 before any billing or key lookup code runs.

#### Why was this change needed?

The old flow turned a model-resolution failure into a misleading key-resolution failure. The new guard keeps the error close to the real cause.

### Bug 3: Missing parent resources bypass ownership checks

**Plan reference**: `.context/attachments/plan.md`
**Impact**: High | **Severity**: Major | **Priority**: P0
**Status**: Fixed

#### What was the problem?

Legacy ownership helpers returned early when the repo, workspace, or session did not exist. That meant some endpoints returned `200 []` instead of `404`, which hid bad IDs and made the API inconsistent.

#### What was changed?

**Before:**
```python
if ws:
    check_owner(ws["owner_email"], get_user_email(request))
```

**After:**
```python
if ws:
    check_owner(ws["owner_email"], get_user_email(request))
else:
    raise HTTPException(status_code=404, detail="Workspace not found")
```

#### How does the code work?

1. The dependency helper still loads the parent row first.
2. If the row exists, ownership is checked as before.
3. If the row is missing, the helper raises a concrete `404` immediately.

#### Why was this change needed?

Authorization and existence are different checks. The API now reports the missing resource explicitly instead of silently pretending the request was valid.

### Bug 4: Repo deletion aborts on one workspace failure

**Plan reference**: `.context/attachments/plan.md`
**Impact**: High | **Severity**: Major | **Priority**: P1
**Status**: Fixed

#### What was the problem?

Deleting a repo iterated over every workspace cleanup with no error handling. If one worktree removal failed, the entire delete request stopped and the repo row remained.

#### What was changed?

**Before:**
```python
for workspace in workspace_rows:
    await delete_workspace(db, workspace["id"])
```

**After:**
```python
for workspace in workspace_rows:
    try:
        await delete_workspace(db, workspace["id"])
    except Exception:
        logger.warning(
            "Failed to delete workspace %s while deleting repo %s",
            workspace["id"],
            repo_id,
            exc_info=True,
        )
```

#### How does the code work?

1. The endpoint still attempts to clean up every workspace first.
2. A failure is logged as a warning with traceback context.
3. The loop continues, and the repo row is still deleted afterward.

#### Why was this change needed?

Repo deletion should be best-effort for per-workspace cleanup, not all-or-nothing. This prevents one broken worktree from blocking the main delete path.

### Bug 7: Backend mypy errors block static verification

**Plan reference**: `.context/attachments/plan.md`
**Impact**: Medium | **Severity**: Major | **Priority**: P2
**Status**: Fixed

#### What was the problem?

The backend had 45 mypy errors across route handlers and service helpers. The biggest patterns were missing return annotations, untyped `dict` usage, nullable values that were treated as strings, and `Any` leaking through JSON or database APIs.

#### What was changed?

Representative fixes:

**Before:**
```python
def list_sessions(workspace_id: str, request: Request) -> list[dict]:
```

**After:**
```python
def list_sessions(
    workspace_id: str,
    request: Request,
) -> list[dict[str, Any]]:
```

#### How does the code work?

1. Route handlers now declare concrete response shapes.
2. Helper functions use typed `sqlite3.Connection` parameters.
3. Sidecar, auth, and container helpers validate or cast dynamic values before returning them.

#### Why was this change needed?

This makes the codebase statically checkable again and prevents the same class of nullability and `Any` bugs from re-entering unnoticed.

### Bug 9: Session tree walks unbounded directory trees

**Plan reference**: `.context/attachments/plan.md`
**Impact**: High | **Severity**: Major | **Priority**: P1
**Status**: Fixed

#### What was the problem?

The session tree endpoint walked the entire workspace recursively and only skipped `.git`. Large folders like `node_modules` or `dist` could make the endpoint slow or memory-heavy.

#### What was changed?

**Before:**
```python
for dirpath, dirnames, filenames in os.walk(workspace_path):
    dirnames[:] = [d for d in dirnames if d != ".git"]
```

**After:**
```python
_EXCLUDED_DIRS = frozenset({...})
_TREE_FILE_LIMIT = 5000

for dirpath, dirnames, filenames in os.walk(workspace_path):
    dirnames[:] = sorted(
        dirname
        for dirname in dirnames
        if dirname not in _EXCLUDED_DIRS
    )
```

#### How does the code work?

1. A helper now owns tree walking.
2. Common generated directories are removed from the traversal list before recursion.
3. The response stops after 5000 files and returns a sorted list.

#### Why was this change needed?

Tree rendering should stay fast even in real repositories. The cap and exclusion list bound both latency and memory use.

### Bug 10: Frontend silently swallows key errors

**Plan reference**: `.context/attachments/plan.md`
**Impact**: Medium | **Severity**: Minor | **Priority**: P3
**Status**: Fixed

#### What was the problem?

Several `catch {}` blocks in the sidebar and session page discarded useful failures. Users had no feedback when repositories failed to load or workspaces failed to create.

#### What was changed?

Critical flows now show inline error messages, and non-critical flows log to `console.error`.

#### How does the code work?

1. Sidebar repository loading stores `repoLoadError`.
2. Workspace loading and creation store `workspaceError`.
3. Session history loading stores `historyError`.
4. Less critical actions still log for debugging without interrupting the UI.

#### Why was this change needed?

Silence makes support and debugging much harder. Users now see actionable feedback for the main failures that block navigation.

### Bug 11: Repo API responses leak `owner_email`

**Plan reference**: `.context/attachments/plan.md`
**Impact**: High | **Severity**: Major | **Priority**: P1
**Status**: Fixed

#### What was the problem?

The backend response model exposed `owner_email`, and the frontend client type accepted it. That leaked unnecessary user identity data through normal repo endpoints.

#### What was changed?

**Before:**
```python
class RepoOut(BaseModel):
    ...
    owner_email: str | None = None
```

**After:**
```python
class RepoOut(BaseModel):
    ...
    custom_prompt: str | None = None
```

#### How does the code work?

1. The Pydantic response model no longer includes `owner_email`.
2. FastAPI filters that field out of serialized repo responses automatically.
3. The frontend `Repo` interface was updated to match.

#### Why was this change needed?

The email was not required by the UI, so exposing it only increased privacy risk.

### Bug 13: Import form assumes every free-form input is `owner/repo`

**Plan reference**: `.context/attachments/plan.md`
**Impact**: Medium | **Severity**: Minor | **Priority**: P3
**Status**: Fixed

#### What was the problem?

If a user entered arbitrary text that was not a URL or a local path, the sidebar always turned it into `https://github.com/<input>`. Invalid inputs were treated as GitHub shorthand even when they were not.

#### What was changed?

**Before:**
```ts
} else {
  body.remote_url = `https://github.com/${value}`;
}
```

**After:**
```ts
} else if (isGithubShorthand(value)) {
  body.remote_url = `https://github.com/${value}`;
} else {
  setError("Enter a GitHub URL, owner/repo, or local path.");
  return;
}
```

#### Why was this change needed?

The form now distinguishes valid GitHub shorthand from unrelated free-form text and gives the user a clear error instead of inventing a bad URL.
