# Code Review: Cloud Runner Registration

Date: 2026-04-25
Branch: feature/byoc-runner

I reviewed the staged BYOC runner changes in the backend runner API, runner
service, runner agent, control database schema, CloudFormation template, and
settings UI. The review covered token handling, public endpoint exposure,
control database migration safety, runner liveness behavior, and regression
coverage.

No blocking issues were found.

The token model is sound for this slice. Yinshi creates high-entropy
registration and bearer tokens, stores only SHA-256 hashes in the control
database, and never logs token values. The two open runner endpoints are still
credential-gated: `/runner/register` needs the one-time token, and
`/runner/heartbeat` needs the runner bearer token. The runner agent also removes
the consumed registration token from its systemd environment file after it has a
stored bearer token.

The API changes are additive. Existing provider settings routes keep their
shape. New runner settings routes use the same FastAPI `detail` error shape as
the rest of the service, and the frontend types mirror the backend response
models.

The database change is safe to roll forward. It adds `user_runners` plus indexes
and a timestamp trigger. Existing tenant rows are left alone, as are current
settings rows and provider records. Existing control databases are upgraded
through `_migrate_control` without dropping data.

The runner liveness path is intentionally conservative. User-facing status is
computed from heartbeat freshness instead of trusting a stale stored status. The
runner agent checks that `/var/lib/yinshi` behaves like writable POSIX storage
before it advertises SQLite support, worktree support, or sidecar capability
flags. If the control plane rejects the stored bearer token, the agent exits
rather than silently pretending to be healthy.

Tests cover the important lifecycle paths: token creation, rejected token reuse,
successful heartbeat, status display after heartbeat, revoked runner behavior,
and missing bearer auth. Frontend coverage verifies that settings can create and
display the runner registration values.

This change stops at registration and heartbeat. Actual dispatch of repo work,
workspace work, session work, or Pi config work to the user-owned runner is still
a later implementation slice. The CloudFormation template installs from the
GitHub repository, so pre-merge AWS testing needs either this branch pushed or an
overridden package install command.
