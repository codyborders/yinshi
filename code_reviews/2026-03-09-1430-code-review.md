## Summary

The codebase implements an SSE-based streaming architecture where the frontend sends prompts via HTTP POST, the backend streams events from a sidecar agent, and the frontend renders rich conversation turns with tool calls, thinking blocks, and inline diffs.

**Key takeaways:**
- SQL injection protection is solid (all queries use parameterization)
- The SSE streaming pipeline works but has performance bottlenecks from synchronous DB calls blocking the async event loop
- The frontend renders correctly but causes excessive React re-renders during streaming
- There is no authorization model -- any authenticated user can access any resource
- Error messages leak internal infrastructure details to the client
- The event format parsing has duplicated logic across multiple files (information leakage in the design sense)

**Finding counts:** 2 Critical, 6 Major, 12 Minor

## Design Findings

### D-1: Information Leakage -- SSE Event Format Knowledge Spread Across 3 Modules (MAJOR)

**WHAT:** The sidecar's event format (field naming, event types, structure) is understood by three separate files:
- `stream.py` constructs SSE events from sidecar output
- `client.ts` defines the `SSEEvent` TypeScript type union
- `useAgentStream.ts` interprets event fields with triple-fallback chains

**HOW:** The sidecar uses camelCase field names (`toolName`, `toolInput`), while the Anthropic API uses different names (`name`, `tool_name`). The frontend handles this with defensive fallback chains:

```typescript
// In useAgentStream.ts -- this pattern appears TWICE (lines 119-124 and 138-142)
const toolName =
  (ev.toolName as string) ||    // sidecar format
  (ev.name as string) ||        // Anthropic format
  (ev.tool_name as string) ||   // snake_case variant
  "unknown";
```

**WHY this matters:** If the sidecar's format changes, you must update all three files in coordination. This violates the "information hiding" principle -- the sidecar's wire format should be normalized in exactly one place.

**Recommendation:** Add a `normalizeSSEEvent()` function in `client.ts` that converts any incoming event to a canonical shape before yielding it from the async generator. Then `useAgentStream.ts` only needs to understand one format.

### D-2: Repeated Streaming Dots Pattern (MINOR)

**WHAT:** The animated "pulsing dots" streaming indicator appears in 3 places with near-identical code:

1. `ChatView.tsx` lines 117-121 (global streaming fallback)
2. `AssistantTurn.tsx` lines 108-112 (in the summary header)
3. `AssistantTurn.tsx` lines 175-179 (when no summary header exists)

**HOW:** Each instance renders 3 `<span>` elements with `animate-pulse`, `rounded-full`, and `bg-blue-400` classes, differing only in size (`h-1.5 w-1.5` vs `h-1 w-1`).

```tsx
// This exact pattern appears 3 times:
<span className="h-1.5 w-1.5 animate-pulse rounded-full bg-blue-400" />
<span className="h-1.5 w-1.5 animate-pulse rounded-full bg-blue-400 [animation-delay:0.2s]" />
<span className="h-1.5 w-1.5 animate-pulse rounded-full bg-blue-400 [animation-delay:0.4s]" />
```

**WHY this matters:** If you change the animation style, color, or timing, you must update 3 places. A small `<StreamingDots size="sm" />` component eliminates the duplication.

### D-3: Duplicate Icon Systems (MINOR)

**WHAT:** `AssistantTurn.tsx` has an emoji-based `TOOL_ICONS` map (lines 16-26) while `ToolCallBlock.tsx` has a full SVG-based `ToolIcon` component (lines 67-185). Both map tool names to visual icons for different purposes.

**HOW:** The emoji map is used in the collapsible turn summary header. The SVG icons are used inside expanded tool call blocks.

```typescript
// AssistantTurn.tsx -- emoji icons
const TOOL_ICONS: Record<string, string> = {

  Edit: "\u{270F}\u{FE0F}",
  Bash: "\u{1F4BB}",

// ToolCallBlock.tsx -- SVG icons (67 lines of JSX)
function ToolIcon({ name }: { name: string }) {
  switch (name) {
    case "Read": return (<svg>...</svg>);
    // ...
  }
```

**WHY this matters:** Two parallel representations of the same concept. If a new tool is added, both must be updated. Consider using the SVG `ToolIcon` component everywhere for consistency.

### D-4: Nonobvious Mutable Array Pattern in useAgentStream (MINOR)

**WHAT:** The `blocks` array in `useAgentStream.ts` is mutated in-place during streaming, then shallow-copied before passing to React state.

**HOW:** The code appends to and mutates `blocks` directly:

```typescript
// Line 54: blocks is a plain mutable array
const blocks: TurnBlock[] = [];

// Line 103: mutating the last element in-place
last.text = (last.text || "") + block.text;

// Line 61: then shallow-copying before React sees it
const snapshot = blocks.map((b) => ({ ...b }));
```

**WHY this matters:** A reader might assume React detects mutations on the original array. Correctness depends entirely on the shallow copy in `upsertTurn()` happening before `setMessages` processes the update. This works but is fragile -- if someone removes the snapshot step, React will silently stop updating the UI because it sees the same array reference.

## Security Findings

### S-1: Full Exception Strings Sent to Client (CRITICAL)

**WHAT:** Raw Python exception messages are forwarded to the browser via SSE events.

**HOW:** When any error occurs during streaming, the catch block serializes the exception message directly:

```python
# stream.py line 176
except Exception as e:
    yield f"data: {json.dumps({'type': 'error', 'error': str(e)})}\n\n"
```

If the sidecar is down, the client might receive:
```json
{"type": "error", "error": "Cannot connect to Unix socket /tmp/yinshi-sidecar-abc123.sock: Connection refused"}
```

This reveals internal socket paths, host names, and infrastructure details.

**WHY this matters:** An attacker can trigger edge cases (e.g., sidecar being down, invalid workspace path) to enumerate internal infrastructure. OWASP classifies this as "Improper Error Handling" and it enables further targeted attacks.

**Recommendation:** Return a generic error message to the client and log the real exception server-side:

```python
except Exception as e:
    logger.error("Sidecar error: session=%s error=%s", session_id, e, exc_info=True)
    yield f"data: {json.dumps({'type': 'error', 'error': 'An internal error occurred'})}\n\n"
```

### S-2: Race Condition on `_active_sessions` Dict (CRITICAL)

**WHAT:** The module-level `_active_sessions` dict is accessed concurrently by streaming generators and the cancel endpoint with no synchronization.

**HOW:** The lifecycle looks like this:

```python
# Line 24: shared mutable state
_active_sessions: dict[str, SidecarClient] = {}

# Line 103: streaming generator writes to it
_active_sessions[session_id] = sidecar

# Line 225: cancel endpoint reads from it (concurrent)
sidecar = _active_sessions.get(session_id)

# Line 196: streaming generator removes from it
_active_sessions.pop(session_id, None)
```

If a user sends two `/prompt` requests for the same session simultaneously:
1. Request A writes `_active_sessions["abc"] = sidecarA`
2. Request B overwrites `_active_sessions["abc"] = sidecarB`
3. Request A finishes, pops `_active_sessions["abc"]` -- removes B's client
4. B's sidecar is now orphaned (never cleaned up, never cancelable)

**WHY this matters:** Orphaned sidecar connections leak file descriptors. The cancel endpoint can cancel the wrong stream. In multi-worker deployments, the dict is per-worker so cancel silently fails.

**Recommendation:** Check `session.status == 'running'` before starting a new stream and reject with 409. Use `asyncio.Lock` for the dict operations.

### S-3: No Authorization -- Any Authenticated User Can Access Any Resource (MAJOR)

**WHAT:** The database schema has no `owner` column. No endpoint checks whether the requesting user owns the resource.

**HOW:** The auth middleware sets `request.state.user_email` from the session cookie, but no endpoint ever checks it:

```python
# stream.py -- session looked up by ID only, no owner check
session = db.execute(
    "SELECT s.*, w.path as workspace_path FROM sessions s "
    "JOIN workspaces w ON s.workspace_id = w.id "
    "WHERE s.id = ?",
    (session_id,),
).fetchone()
```

User A can read User B's conversations, delete User B's repos, or send prompts to User B's sessions -- just by knowing the UUID.

**WHY this matters:** This is horizontal privilege escalation. In a multi-user deployment, all data is effectively public to any authenticated user.

---

### S-4: No CSRF Protection on State-Mutating Endpoints (MAJOR)

**WHAT:** The session cookie uses `samesite="lax"` but there is no CSRF token mechanism. The frontend sends `credentials: "include"` on all fetch requests.

**HOW:** A malicious website could craft a form POST that targets Yinshi endpoints. While `samesite=lax` blocks cross-site POSTs from subresource requests, it does not protect against top-level navigations in all browsers.

**WHY this matters:** An attacker's website could create repos, delete data, or send prompts on behalf of an authenticated user who visits the page.

**Recommendation:** Require a custom header like `X-Requested-With: XMLHttpRequest` on all mutating endpoints. Browsers prevent cross-origin requests from setting custom headers without CORS preflight approval.

---

### S-5: `git clone` with Unsanitized URL (MAJOR)

**WHAT:** User-supplied `remote_url` is passed directly to `git clone` without scheme validation.

**HOW:**

```python
# git.py line 69
await _run_git(["clone", url, dest])
```

Git's `ext::` protocol can execute arbitrary commands. While `create_subprocess_exec` avoids shell injection, git itself interprets the URL.

**WHY this matters:** A user could submit `ext::sh -c 'cat /etc/passwd > /tmp/leaked'` as a repo URL.

**Recommendation:** Validate that `remote_url` starts with `https://` or `ssh://`. Reject `ext::`, `file://`, and URLs starting with `-`.

---

### S-6: No Input Length Validation on Prompts (MINOR)

**WHAT:** The `PromptRequest` Pydantic model accepts arbitrarily long `prompt` strings.

```python
class PromptRequest(BaseModel):
    prompt: str           # No max_length constraint
    model: str | None = None
```

**WHY this matters:** A multi-megabyte prompt could exhaust server memory, bloat the database, and overwhelm the sidecar agent. Add `Field(..., max_length=100000)`.

## Performance Findings

### P-1: Synchronous SQLite Calls Block the Async Event Loop (MAJOR)

**WHAT:** Every `get_db()` call inside `stream.py`'s async generator performs synchronous `sqlite3` operations that block the entire event loop.

**HOW:** During a single streaming turn, `get_db()` is called approximately:
- 2 times before streaming starts (session lookup + user message save)
- 1 time per assistant chunk (incremental persistence)
- 1 time on result finalization
- 1 time in the finally block

For a turn with 50 assistant events, that is ~54 synchronous connection open/close cycles, each blocking the event loop for 1-5ms.

```python
# stream.py line 135 -- called on EVERY assistant chunk
with get_db() as db:  # synchronous! blocks event loop
    if assistant_msg_id is None:
        db.execute("INSERT INTO messages ...")
    else:
        db.execute("UPDATE messages SET content = ? ...")
    db.commit()
```

**WHY this matters:** All other concurrent requests (including other streaming sessions) are blocked while SQLite I/O runs. With 5 concurrent sessions, the event loop is blocked 10-25% of the time.

**Recommendation:** Wrap synchronous DB calls in `asyncio.to_thread()`, or switch to `aiosqlite`.

---

### P-2: React Re-renders on Every SSE Event (MAJOR)

**WHAT:** `upsertTurn()` calls `setMessages()` on every single SSE event, triggering a full React re-render of the entire message list.

**HOW:** Each call to `upsertTurn()`:
1. Filters all blocks to extract text content -- O(n)
2. Shallow-copies every block -- O(n)
3. Calls `setMessages(prev => ...)` -- triggers React reconciliation
4. Inside the updater, runs `findIndex` -- O(m) where m = total messages
5. Copies the entire messages array -- O(m)

```typescript
// useAgentStream.ts line 56 -- called on EVERY SSE event
function upsertTurn(done = false) {
  const allText = blocks.filter((b) => b.type === "text").map((b) => b.text || "").join("");
  const snapshot = blocks.map((b) => ({ ...b }));
  setMessages((prev) => {
    // ...full array copy + findIndex on every event
  });
}
```

With 50-200 SSE events per turn, this causes 50-200 full React render cycles.

**WHY this matters:** Visible jank and dropped frames during fast streaming, especially on mobile. Each re-render cascades through `ChatView` -> all `AssistantTurn` components -> all `ToolCallBlock` components.

**Recommendation:**
- Throttle `upsertTurn()` to at most every 50-100ms using `requestAnimationFrame`
- Add `React.memo()` to `AssistantTurn`, `ToolCallBlock`, and `MessageBubble`
- Use `useMemo` for `turnSummary()` in `AssistantTurn`

### P-3: O(n^2) String Rebuild on Every Chunk (MINOR)

**WHAT:** Every time an assistant chunk arrives, the backend re-joins the entire `assistant_chunks` list.

**HOW:**

```python
# stream.py line 133 -- on EVERY chunk event
accumulated = "".join(assistant_chunks)
```

If a turn produces `k` chunks totaling `N` characters, the total string copying work is O(k * N) -- quadratic. For a 50KB response across 500 chunks, total characters processed is ~12.5MB.

**WHY this matters:** CPU overhead scales poorly with response length. The simple fix is to maintain a running string instead of re-joining the list every time.

---

### P-4: Auto-Scroll Causes Layout Thrashing (MINOR)

**WHAT:** The auto-scroll `useEffect` in `ChatView.tsx` reads `scrollHeight` and writes `scrollTop` on every `messages` state change.

```typescript
// ChatView.tsx line 24-29
useEffect(() => {
  const el = scrollRef.current;
  if (el) {
    el.scrollTop = el.scrollHeight;  // forces layout reflow
  }
}, [messages]);
```

**WHY this matters:** Reading `scrollHeight` forces the browser to calculate layout. Writing `scrollTop` forces another layout pass. This runs 50-200 times per streaming turn. It also always scrolls to bottom even if the user has scrolled up to review earlier content.

**Recommendation:** Use `requestAnimationFrame` to batch scroll updates. Track whether the user is near the bottom before auto-scrolling.

---

### P-5: Missing React.memo on Child Components (MINOR)

**WHAT:** `AssistantTurn` and `ToolCallBlock` do not use `React.memo()`. Every time `ChatView` re-renders (on every SSE event), all message components re-render even if their props have not changed.

**WHY this matters:** Could reduce React reconciliation work by 70-90% during streaming. Only the currently-streaming message actually changes -- all previous messages have stable props.

---

### P-6: Duplicate Database Query in prompt_session (MINOR)

**WHAT:** Two separate `get_db()` calls fetch overlapping data from the same tables.

```python
# First query (line 50): fetches session + workspace path
session = db.execute(
    "SELECT s.*, w.path as workspace_path FROM sessions s "
    "JOIN workspaces w ON s.workspace_id = w.id WHERE s.id = ?", ...
)

# Second query (line 81): fetches workspace id, name, branch
workspace = db.execute(
    "SELECT w.id, w.name, w.branch FROM workspaces w "
    "JOIN sessions s ON s.workspace_id = w.id WHERE s.id = ?", ...
)
```

**WHY this matters:** The first query could easily include `w.id, w.name, w.branch` and eliminate the second round trip entirely.

## Code Quality Findings

### Q-1: Synchronous `get_db()` in Async Handlers Violates Project Conventions (MAJOR)

The project's `CLAUDE.md` says "Use async/await for I/O-bound operations." Every `get_db()` call from async route handlers blocks the event loop. This is both a performance issue (P-1) and a convention violation.

---

### Q-2: `init_db()` Leaks Connection on Exception (MAJOR)

**WHAT:** `init_db()` opens a connection manually without a context manager:

```python
# db.py lines 89-91
def init_db() -> None:
    conn = sqlite3.connect(settings.db_path)
    conn.executescript(SCHEMA_SQL)
    conn.close()  # never reached if executescript raises
```

**WHY this matters:** If `executescript` raises (e.g., invalid SQL), `conn.close()` is never called. The fix is trivial:

```python
def init_db() -> None:
    with get_db() as conn:
        conn.executescript(SCHEMA_SQL)
```

---

### Q-3: Bare `except Exception` Swallows Programming Errors (MINOR)

**WHAT:** The streaming generator catches all exceptions:

```python
# stream.py lines 171-176
except Exception as e:
    yield f"data: {json.dumps({'type': 'error', 'error': str(e)})}\n\n"
```

This catches `TypeError`, `KeyError`, and other bugs in the chunk-processing logic, converting them silently into SSE error events instead of letting them surface during development.

**Recommendation:** Catch specific expected exceptions (`ConnectionError`, `OSError`, known sidecar errors) and let unexpected exceptions propagate.

---

### Q-4: Missing Type Hints on All Test Fixtures and Helpers (MINOR)

**WHAT:** No test fixture or helper function has return type annotations:

```python
# conftest.py -- no return types
def tmp_dir(tmp_path):        # -> str
def db_path(tmp_path):        # -> str
def db(db_path, monkeypatch): # -> Iterator[sqlite3.Connection]
def git_repo(tmp_path):       # -> str

# test_api.py -- no return types
def client(db_path, monkeypatch):  # -> Iterator[TestClient]
def session_id(client, git_repo):  # -> str
```

The project's `PYTHON.md` requires type hints on function signatures.

### Q-5: Unused Fixtures in conftest.py (MINOR)

**WHAT:** Two fixtures are defined but never used:
- `mock_sidecar` (line 39) -- tests use `_make_mock_sidecar()` helper instead
- `tmp_dir` (line 11) -- trivially wraps `str(tmp_path)`, no test references it

**WHY this matters:** Dead code creates confusion about which fixtures to use. Remove both.

### Q-6: Test Setup Duplication (MINOR)

**WHAT:** The "create repo -> create workspace -> create session" pattern is repeated in 5+ test functions manually, despite the `session_id` fixture existing.

**WHY this matters:** Tests like `test_first_prompt_updates_workspace_name` need access to intermediate IDs (repo_id, workspace_id). A fixture returning a named tuple `(repo_id, workspace_id, session_id)` would eliminate the boilerplate.

### Q-7: `AsyncGenerator` Imported from `typing` Instead of `collections.abc` (MINOR)

```python
# stream.py line 11 -- legacy import
```

## Simplification Findings

### F-1: Extract "Append or Create Block" Helper in useAgentStream.ts (MAJOR)

**WHAT:** The same pattern appears 4 times:

```typescript
// This pattern repeats for: text in assistant, thinking in assistant,
// text in content_block_delta, thinking in content_block_delta
const last = blocks[blocks.length - 1];
if (last && last.type === "text") {
  last.text = (last.text || "") + newText;
} else {
  blocks.push({ id: nextId(), type: "text", text: newText });
}
```

**Recommendation:** Extract to a helper:

```typescript
function appendOrCreate(blocks: TurnBlock[], type: "text" | "thinking", text: string) {
  const last = blocks[blocks.length - 1];
  if (last && last.type === type) {
    last.text = (last.text || "") + text;
  } else {
    blocks.push({ id: nextId(), type, text });
  }
}
```

### F-3: `toolSummary` Returns String That Gets Regex-Parsed Back (MINOR)

**WHAT:** `toolSummary()` in `ToolCallBlock.tsx` returns `diffStats` as a formatted string like `"+5 -3"`. The rendering code then uses regex to extract the numbers back out:

```tsx
// toolSummary returns: { diffStats: "+5 -3" }
// ToolCallBlock renders:
<span className="text-green-500">
  +{summary.diffStats.match(/\+(\d+)/)?.[1] || 0}
</span>
```

**Recommendation:** Return `{ added: number, removed: number }` instead of a formatted string.
