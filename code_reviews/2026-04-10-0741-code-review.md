# Code Review

## Summary

Reviewed the default per-user container isolation changes against `origin/main`, focusing on container lifecycle, tenant sidecar routing, repository materialization, and provider OAuth behavior. I found three major regressions in the new container-default path.

## Design Findings

### MAJOR: The idle reaper can destroy containers that still hold live prompt or OAuth state

`ContainerManager` only tracks a single `last_activity` timestamp, and that timestamp is refreshed only when a container is created/reused or when a caller explicitly invokes `touch()`. The new tenant prompt flow does not refresh that timestamp while a stream is in progress, and the new OAuth routes do not hold containers open while the user is away completing the browser step. At the same time, the sidecar keeps both active sessions and active OAuth flows only in memory.

In practice, any prompt that runs longer than `container_idle_timeout_s` can be reaped mid-turn, and any OAuth flow that takes longer than that timeout can disappear before the callback arrives. This is a behavioral regression introduced by moving tenant OAuth and prompt execution into per-user containers without giving the reaper a notion of "busy" containers.

## Code Quality Findings

### MAJOR: Container readiness can be satisfied by a stale socket file from the previous sidecar

The container startup path reuses the per-user socket directory and declares success as soon as `os.path.exists(socket_path)` becomes true. That is not a reliable readiness signal for a Unix socket that lives on a reused bind mount. If the previous sidecar left a stale `sidecar.sock` behind, `_wait_for_socket()` can return immediately before the new sidecar has actually rebound the socket.

The sidecar process itself explicitly unlinks any pre-existing socket path during startup, which makes the race more obvious: the backend can observe the old file and return "ready" before the new process has finished its own cleanup and listen step.

### MAJOR: Catalog and provider OAuth requests now return raw 500s for routine tenant-sidecar failures

The new catalog and provider OAuth routes convert `ensure_container()` failures into a generic `503`, but they do not do the same for failures that happen one step later while connecting to or querying the tenant sidecar. `create_sidecar_connection()` can raise `SidecarNotConnectedError`, and the catalog / OAuth client methods can raise `SidecarError`; those exceptions currently bubble out as unhandled `500` responses.

That gap matters more after this branch because those routes now depend on short-lived per-user containers by default, so transient tenant-sidecar startup or restart issues become much more common than in the old shared-sidecar path.

## Simplification Findings

The lifecycle policy is currently spread across route handlers via `touch_tenant_container()`. This is exactly the kind of stateful policy that should live inside `ContainerManager`, because callers cannot reliably infer every moment when a container is still "in use." A deeper container module would own readiness, busy/idle tracking, and error translation instead of asking each endpoint to remember to touch the container at the right times.
