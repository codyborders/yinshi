# Yinshi Code Review Report

**Date:** 2026-03-09
**Reviewer:** Claude Code (multi-agent review with design analysis, security audit, performance review, Python quality review, and code simplification)
**Scope:** Full-stack -- Python/FastAPI backend, Node.js sidecar, React/TypeScript frontend

---

## Summary

Yinshi is a greenfield web application that orchestrates parallel coding agents across isolated git worktrees. The architecture is clean: FastAPI backend, Node.js pi sidecar via Unix socket, React frontend with WebSocket streaming. The codebase is concise and well-structured for a first implementation.

However, the review uncovered **4 Critical**, **5 High**, and **8 Medium/Minor** issues across security, performance, architecture, and code quality. The most severe findings are: a **singleton sidecar socket that corrupts data under concurrency**, **synchronous SQLite blocking the async event loop**, **no authorization on any endpoint**, **unauthenticated WebSocket access**, and a **live API key in the `.env` file**.

## Design Findings

### D1. Singleton Sidecar Socket is Not Concurrency-Safe (CRITICAL)

**File:** `backend/src/yinshi/services/sidecar.py:85-117`

**What:** The `SidecarClient` is a singleton -- one `asyncio.StreamReader`/`StreamWriter` pair shared by all concurrent WebSocket connections. The `query()` method sends a request and then reads responses line-by-line, skipping lines whose `id` does not match the current session:

```python
# sidecar.py:100-116 (current code)
while True:
    msg = await self._read_line()
    if msg is None:
        raise SidecarError("Sidecar connection lost")
    if msg.get("id") != session_id:
        continue  # <-- discards messages for other sessions
    yield msg
    # ...
```

**How it works:** When two WebSocket sessions call `sidecar.query()` concurrently, both coroutines read from the same `self._reader`. Whichever coroutine's `readline()` returns first consumes the line. If session A's coroutine reads a line belonging to session B, it silently discards it (`continue`). That line is permanently lost -- session B never sees it.

**Why this matters:** This is a correctness bug, not just a performance issue. With 2+ concurrent sessions, agent responses will have missing text chunks, incomplete tool calls, or will hang indefinitely waiting for a "result" terminator that was consumed and discarded by another session's coroutine.

**Fix:** Open a separate Unix socket connection per active session (the sidecar already supports multiple connections -- `sidecar.js:68`), or add a background demultiplexer that routes messages to per-session `asyncio.Queue` instances.

### D2. Synchronous SQLite in Async Handlers Blocks the Event Loop (CRITICAL)

**Files:** `backend/src/yinshi/db.py:69-76`, all API route files, `backend/src/yinshi/api/ws.py`

**What:** Every route handler is declared `async def` but calls `sqlite3.connect()`, `execute()`, `commit()`, and `close()` -- all synchronous, blocking operations:

```python
# repos.py:20-23 (current pattern repeated everywhere)
async def list_repos() -> list[dict]:
    db = get_db()             # blocks event loop
    rows = db.execute(...)    # blocks event loop
    db.close()                # blocks event loop
    return [dict(r) for r in rows]
```

**How it works:** In FastAPI, `async def` endpoints run on the main asyncio event loop thread. Synchronous calls in these handlers block the entire thread -- no other coroutine can make progress (including other HTTP requests and WebSocket frames) until each blocking call returns.

The WebSocket handler (`ws.py`) is the worst offender: it holds a synchronous `db` connection open for the entire WebSocket lifetime (potentially hours) and calls `db.execute()`/`db.commit()` on every streaming event.

**Why this matters:** With a single user, SQLite operations are sub-millisecond and invisible. With 5+ concurrent WebSocket sessions, each streaming and committing on every event, the event loop will be blocked hundreds of milliseconds per second aggregate. API endpoints will see p99 latencies spike to 50-200ms.

FastAPI has a built-in solution: declare handlers as plain `def` (not `async def`), and FastAPI automatically runs them in a threadpool. By using `async def` with blocking I/O, you get the worst of both worlds.

**Fix:** Either:
1. Change all handlers that only do synchronous SQLite to plain `def` (not `async def`), OR
2. Use `asyncio.to_thread()` to wrap synchronous DB calls, OR
3. Switch to `aiosqlite`

### D3. Database Connection Pattern -- No Context Manager, Leak-Prone (MAJOR)

**File:** `backend/src/yinshi/db.py:69-76`

**What:** `get_db()` returns a raw `sqlite3.Connection`. Every caller must manually call `db.close()`, and several handlers risk connection leaks on exception paths:

```python
# repos.py:17-23 -- no try/finally, exception = leak
async def list_repos() -> list[dict]:
    db = get_db()
    rows = db.execute("SELECT * FROM repos ORDER BY created_at DESC").fetchall()
    db.close()  # never reached if execute() throws
    return [dict(r) for r in rows]
```

This pattern is repeated ~10 times across `repos.py`, `workspaces.py`, `sessions.py`, and `ws.py`. Some handlers use `try/finally`, some don't. The inconsistency makes bugs likely.

**Why this matters:** A connection that is never closed holds a file handle to the SQLite database. Under WAL mode, enough leaked connections can prevent checkpointing and cause the WAL file to grow unboundedly.

**Fix:** Make `get_db()` a context manager or a FastAPI dependency with `yield`:

```python
from contextlib import contextmanager

@contextmanager
def get_db():
    conn = sqlite3.connect(settings.db_path)
    conn.row_factory = sqlite3.Row
    conn.execute("PRAGMA foreign_keys = ON")
    try:
        yield conn
    finally:
        conn.close()
```

Then all callers use `with get_db() as db:` -- impossible to forget cleanup.

### D4. WebSocket Handler is a 130-Line Monolith (MAJOR)

**File:** `backend/src/yinshi/api/ws.py:16-138`

**What:** The `websocket_agent` function handles:
- WebSocket lifecycle
- Session validation via DB lookup
- Sidecar warmup/connection
- User message parsing and dispatch
- Database writes for user messages, assistant messages, and session status
- Streaming event accumulation and forwarding
- Error handling for all of the above

**Why this matters:** This function is untestable without mocking WebSocket, database, and sidecar simultaneously. The deeply nested event processing logic (lines 90-117) is fragile -- navigating `event["data"]["message"]["content"][i]["text"]` with minimal validation.

Additionally, two separate `db.commit()` calls on lines 80 and 87 should be a single transaction (inserting user message + updating session status are one logical operation).

---

### D5. Duplicated "Fetch or 404" Pattern (MINOR)

**Files:** `repos.py`, `sessions.py`, `workspaces.py`

The pattern "fetch a row by ID, raise 404 if not found" appears 7+ times across route files. A small helper would eliminate this duplication:

```python
def fetch_or_404(db, table, id_value, detail="Not found"):
    row = db.execute(f"SELECT * FROM {table} WHERE id = ?", (id_value,)).fetchone()
    if not row:
        raise HTTPException(status_code=404, detail=detail)
    return dict(row)
```

This hook provides `listSessions`, `createSession`, `getSession` -- but `WorkspaceView.tsx` performs the same operations inline without using the hook. The hook is defined, exported, but never imported or called anywhere.

### D7. Model Name "minimax" Leaked Across All Layers (MINOR)

Changing the default model requires coordinated edits across all layers. This should be a single configuration value.

## Security Findings

### C1. Live API Key in `.env` File (CRITICAL)

**File:** `.env`, line 27

```
MINIMAX_API_KEY=sk-cp-lp5GXzTLwkQgaEml-...
```

A live MiniMax API key is present in the `.env` file. While `.env` is in `.gitignore`, if this file was ever committed to version history, the key is permanently leaked.

**Fix:** Immediately rotate the key. Verify with `git log -p --all -S 'sk-cp-lp5G'`. Use `.env.example` for templates with empty placeholders.

---

### C2. Weak Default `secret_key` -- Session Token Forgery (CRITICAL)

**File:** `backend/src/yinshi/config.py:23`

```python
secret_key: str = "change-me-in-production"
```

This key is used by `itsdangerous` to sign session tokens. If deployed with the default, any attacker can forge valid session cookies for any email address.

**Fix:** Remove the default (make it required), or fail-fast on startup if the default is detected. Generate a strong key: `python -c "import secrets; print(secrets.token_hex(32))"`.

---

### C3. No Authorization -- Any User Can Access Any Resource (CRITICAL)

**Files:** All API endpoints in `repos.py`, `workspaces.py`, `sessions.py`, `ws.py`

No endpoint checks whether the authenticated user owns the resource. The database schema has no `user_id` column on any table. Once authenticated, any user can read, modify, and delete any other user's repositories, workspaces, sessions, and messages.

**Fix:** Add `user_id TEXT NOT NULL` to the `repos` table. Filter all queries by the authenticated user's identity.

---

### H1. WebSocket Endpoint Has No Authentication (HIGH)

**File:** `backend/src/yinshi/auth.py:67-69`, `backend/src/yinshi/api/ws.py`

The `AuthMiddleware` explicitly skips WebSocket upgrade requests (comment says "auth checked in WS handler"), but the WebSocket handler performs NO authentication check. An attacker who knows a session ID can connect and send prompts to the coding agent -- effectively **unauthenticated remote code execution** through the agent's tools.

**Fix:** Extract the session cookie from the WebSocket handshake and verify it:

```python
token = websocket.cookies.get("yinshi_session")
email = verify_session_token(token)
if not email:
    await websocket.close(code=4001, reason="Not authenticated")
    return
```

---

### H2. Path Traversal via `local_path` (HIGH)

**File:** `backend/src/yinshi/api/repos.py:31-37`

The `local_path` parameter accepts any absolute path with no restriction. An attacker can register any git repository on the server's filesystem, then use the agent's coding tools for arbitrary file read/write.

**Fix:** Restrict `local_path` to an allowed base directory. Use `os.path.realpath()` to resolve symlinks.

---

### H3. Authentication Disabled When OAuth Not Configured (HIGH)

**File:** `backend/src/yinshi/auth.py:59-61`

When `GOOGLE_CLIENT_ID` is empty (as shipped in the default `.env`), the `AuthMiddleware` passes every request through with no authentication. Any deployment that forgets OAuth configuration is fully open.

**Fix:** Require an explicit `DISABLE_AUTH=true` flag rather than inferring from empty credentials. Refuse to start in production without OAuth configured.

---

### H4. Session Cookie Missing `Secure` Flag (HIGH)

**File:** `backend/src/yinshi/api/auth_routes.py:38-44`

The cookie is set with `httponly=True` and `samesite="lax"` (good), but missing `secure=True`. Over HTTP, the session token is exposed to network sniffing.

**Fix:** Add `secure=not settings.debug`.

---

### H5. Error Messages Leak Internal State (HIGH)

**Files:** `ws.py:53-56`, `ws.py:131-135`, `workspaces.py:52`

Raw Python exception messages are forwarded directly to clients, exposing internal paths, stack traces, and system state.

**Fix:** Return generic error messages to clients. Log detailed errors server-side only.

---

### M1. No CSRF Protection (MEDIUM)

State-changing endpoints (POST, PATCH, DELETE) have no CSRF token mechanism. `SameSite=Lax` provides some protection but not complete.

---

### M2. No Rate Limiting (MEDIUM)

No rate limiting on any endpoint. Each prompt incurs API costs, and the WebSocket has no connection limits.

---

### M3. No Security Headers (MEDIUM)

No `Content-Security-Policy`, `X-Frame-Options`, `X-Content-Type-Options`, or `Strict-Transport-Security` headers.

### M4. Sidecar Socket in World-Readable `/tmp` (LOW)

**File:** `config.py:26`

The Unix socket at `/tmp/yinshi-sidecar.sock` is accessible to any local user, bypassing all application authentication.

**Fix:** Use `$XDG_RUNTIME_DIR` or set socket permissions to 0600.

### M5. OpenAPI Docs Exposed Without Auth (LOW)

**File:** `auth.py:53`

`/docs` and `/openapi.json` are in `OPEN_PATHS`, exposing the full API schema to unauthenticated users.

## Performance Findings

### P1. Singleton Sidecar -- Concurrent Sessions Will Lose Data (CRITICAL)

(Same as D1 above.) The singleton `SidecarClient` cannot safely handle multiple concurrent sessions. Messages will be silently dropped, causing incomplete agent responses.

---

### P2. Synchronous SQLite Blocks Event Loop (CRITICAL)

(Same as D2 above.) All `async def` handlers call synchronous `sqlite3` operations, blocking the entire event loop.

---

### P3. No Connection Pooling (MAJOR)

**File:** `backend/src/yinshi/db.py:69-76`

`get_db()` creates a new `sqlite3.Connection` per call. Each call incurs file I/O (opening the database, reading WAL header, setting PRAGMAs). The WAL PRAGMA in `get_db()` is redundant -- WAL mode persists once set.

---

### P4. WebSocket Holds DB Connection for Entire Lifetime (MAJOR)

**File:** `backend/src/yinshi/api/ws.py:31`

The `db` connection is opened at WebSocket connect and held open until disconnect (potentially hours). This wastes file handles and can prevent SQLite WAL checkpointing.

**Fix:** Open short-lived connections only when needed (for inserts/queries within the event loop).

---

### P5. Frontend Re-renders -- Message Array Copies on Every Event (MINOR)

**File:** `frontend/src/hooks/useWebSocket.ts:65-80`

On every streaming text delta, the hook searches the entire message array for a matching ID, then creates a full copy of the array. For long conversations (hundreds of messages), this becomes expensive.

**Fix:** Use a `Map` or `useReducer` for O(1) message updates instead of linear scans with array spreading.

## Code Quality Findings

### Q1. `async def` Route Handlers That Do No Async Work (CRITICAL)

Every route handler in `repos.py`, `sessions.py`, and `workspaces.py` is declared `async def` but performs only synchronous operations. FastAPI treats `async def` differently from `def` -- async handlers run on the event loop, while sync handlers are automatically run in a threadpool.

**Fix:** Change these to plain `def` (since they only do synchronous SQLite), or use `asyncio.to_thread()`.

---

### Q2. f-String SQL Column Names in `update_repo` (MAJOR)

**File:** `backend/src/yinshi/api/repos.py:83-87`

```python
set_clause = ", ".join(f"{k} = ?" for k in updates)
```

Column names come from `body.model_dump()` keys. Currently safe because Pydantic constrains the keys, but this pattern is fragile and violates the "use parameterized queries" principle.

**Fix:** Whitelist allowed column names explicitly:
```python
ALLOWED_COLUMNS = {"name", "custom_prompt"}
updates = {k: v for k, v in updates.items() if k in ALLOWED_COLUMNS}
```

---

### Q3. Type Hint Inconsistencies (MINOR)

- `auth_routes.py`: `login`, `callback`, `me`, `logout` -- all missing return type annotations
- `auth.py:55`: `dispatch` method has `# type: ignore[override]` with no type for `call_next`
- Inconsistent use of `Optional[str]` vs `str | None` across the codebase (models.py uses `Optional`, git.py uses `|`)

---

### Q4. Unused Import and Variable (MINOR)

- `auth_routes.py:28`: `settings = get_settings()` is called in `callback()` but never used
- `repos.py:5`: `from pathlib import Path` is imported but never used
- `git.py:53,76`: `from pathlib import Path` imported inside two functions -- should be a top-level import

---

### Q5. Redundant Second `useEffect` in `useWebSocket.ts` (MINOR)

**File:** `frontend/src/hooks/useWebSocket.ts:126-129`

The second `useEffect` runs on `[sessionId]` and sets `connected` from `socketRef.current.connected`. The first `useEffect` already manages `connected` state via the socket's event listener. By the time the second effect runs, the socket may not have connected yet, so it reads `false` anyway. This effect is dead code.

## Simplification Findings

### S1. Database Context Manager Would Eliminate 10+ `try/finally` Blocks

Converting `get_db()` to a context manager (or FastAPI `Depends` with `yield`) would remove every manual `db.close()` call and every `try/finally` block wrapping database access across all route files.

---

### S2. Multiple `db.commit()` Calls Should Be Batched

In `ws.py`, user message INSERT (line 80) and session status UPDATE (line 87) are committed separately. Same for assistant message save (line 116) and session status reset (line 127). These are single logical operations and should be one commit each.

---

### S3. `deriveRepoName` Called Twice in Import Form

**File:** `frontend/src/components/RepoList.tsx:162-164`

In the `else` branch of `handleSubmit`, `deriveRepoName(value)` is called on line 163 when it was already called on line 154 (`body.name`). The second call is redundant since `body.name` already has the derived name.

---

### S4. Inline `from pathlib import Path` in Two Functions

**File:** `backend/src/yinshi/services/git.py:53,76`

`from pathlib import Path` is imported inside both `clone_repo` and `create_worktree`. This should be a single top-level import.

---

### S5. Redundant WAL PRAGMA Per Connection

**File:** `backend/src/yinshi/db.py:75`

`conn.execute("PRAGMA journal_mode = WAL")` is called on every `get_db()` connection. WAL mode is persistent once set during `init_db()` -- this per-connection call is unnecessary overhead.
