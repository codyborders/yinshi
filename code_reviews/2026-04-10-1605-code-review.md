Summary

This review covers commit `9c9eb52` (`Use GitHub app tokens for git auth`).

The change moves GitHub write access away from user-managed SSH keys and toward short-lived GitHub App installation tokens. That is the right product direction. However, the current sidecar injection path exposes those tokens to the model-controlled bash environment in ways that make credential exfiltration practical.

Design Findings

- The runtime credential logic is split cleanly between backend token resolution and sidecar execution, which is a good direction.
- The main design problem is information leakage in the sidecar layer: the token injection mechanism is implemented as a generic bash environment mutation, so the secret is now available to any matching bash command instead of only the git subprocess that needs it.

Security Findings

1. CRITICAL: The model can exfiltrate the GitHub installation token directly by asking for any bash command that contains the word `git` and also prints the environment. The spawn hook injects `YINSHI_GIT_TOKEN` into any command whose text matches `/\bgit\b/`, not just actual `git` invocations, and the sidecar forwards the original bash command back to the UI as a normal tool use. A prompt can therefore cause commands like `printf git; env | grep YINSHI_GIT_TOKEN` or `echo git && python -c "import os; print(os.environ['YINSHI_GIT_TOKEN'])"` to succeed and reveal the token in tool output. References: [sidecar.js](/Users/user/conductor/workspaces/yinshi/charlotte/sidecar/src/sidecar.js#L118), [sidecar.js](/Users/user/conductor/workspaces/yinshi/charlotte/sidecar/src/sidecar.js#L141), [sidecar.js](/Users/user/conductor/workspaces/yinshi/charlotte/sidecar/src/sidecar.js#L154), [sidecar.js](/Users/user/conductor/workspaces/yinshi/charlotte/sidecar/src/sidecar.js#L984)

2. CRITICAL: The askpass helper lives at a fixed writable path inside the same container, so the model can replace it before running a git command and steal the token that the hook injects later. The file is created once at `/tmp/yinshi-git-askpass.sh`, and subsequent git commands trust that path blindly. Because the agent has bash/edit/write capability in the same runtime, it can overwrite the script and then trigger a git command to force Git to execute attacker-controlled code with `YINSHI_GIT_TOKEN` in the environment. Performance Findings

- No material performance regressions stood out in this change set. The additional token lookup is small compared to prompt startup and is cached server-side by installation id.

Code Quality Findings

- The backend contract is reasonable and covered by tests.
- The risky part is not correctness under normal use. It is that the sidecar trust boundary is too wide for a secret-carrying mechanism.

Simplification Findings

- The sidecar should not expose a long-lived environment variable to a general-purpose bash tool at all.
- A safer design is to narrow credential scope to the exact subprocess that needs it:
  - either introduce a dedicated git execution tool or sidecar RPC for authenticated git operations,
  - or mutate only the concrete `git` argv execution path rather than shell text, with a per-invocation temporary askpass script in a private directory and no shared token-bearing environment visible to arbitrary commands.
