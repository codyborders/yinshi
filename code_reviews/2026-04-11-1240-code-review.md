## Summary

Reviewed commit `3c0a62f` ("Explicit stop and steering for live agent runs") across the backend stream endpoint, the new run coordinator, the sidecar cancellation path, and the frontend streaming hook.

I found two critical regressions and one major behavioral regression in the new stop/steer flow. I also found one data-correctness issue in the new `turn_status` persistence path. I did not apply code changes in this review.

## Design Findings

- MAJOR: The stop/steer state machine is now split across three modules with no single owner of the invariant that "a queued steering prompt must run after the current turn ends." The frontend hook only replays the queued prompt after a `cancelled` event, while the backend and sidecar can also end a turn with `result` or `error`. That split responsibility is what causes the silent prompt drop described below. Relevant code: `frontend/src/hooks/useAgentStream.ts:266-273`, `backend/src/yinshi/api/stream.py:537-541`, `sidecar/src/sidecar.js:946-960`.

## Code Quality Findings

- CRITICAL: `sidecar/src/sidecar.js` declares `let entry = ...` inside the `try` block (`sidecar/src/sidecar.js:835`) and then reads `entry` from the sibling `catch` block (`sidecar/src/sidecar.js:946`). In JavaScript, `let` is block-scoped, so any cancellation or prompt failure path will throw `ReferenceError: entry is not defined` before the sidecar can emit the intended `cancelled` or `error` event. That breaks the new stop flow at runtime.

- CRITICAL: Authenticated requests use the per-user database from `get_db_for_request()` (`backend/src/yinshi/api/deps.py:30-42`), but the per-user schema in `backend/src/yinshi/tenant.py:75-83` still defines `messages` without the new `turn_status` column. The new stream persistence code writes `turn_status` unconditionally (`backend/src/yinshi/api/stream.py:597-599` and `backend/src/yinshi/api/stream.py:633-641`), so tenant-mode prompts that persist assistant output will fail with SQLite column errors. The legacy migration copier in `backend/src/yinshi/services/accounts.py:137-150` also omits `turn_status`, so even migrated history would lose the new field.

- MAJOR: The new frontend steering logic silently drops the queued prompt unless the current stream emits an explicit `cancelled` event. In `frontend/src/hooks/useAgentStream.ts:266-273`, `queuedPromptRef` is cleared on every completion, but the queued prompt is only replayed when `wasCancelledRef.current` is true. If the current run finishes normally after the user presses steer, or if the cancel request fails or arrives too late, the user's follow-up prompt disappears. The previous implementation always replayed the queued prompt from `finally`, which avoided this race.

- MINOR: Sidecar `error` events are forwarded to the client in `backend/src/yinshi/api/stream.py:605-607`, but `turn_status` is never changed from `"completed"` on that path before the final DB flush at `backend/src/yinshi/api/stream.py:633-641`. As a result, failed turns delivered as explicit sidecar error events will be stored as completed, which makes the new `turn_status` column unreliable.

## Simplification Findings

- `backend/src/yinshi/services/run_coordinator.py:12-22` stores `assistant_msg_id`, `accumulated`, and `chunk_count` on `RunRecord`, but the reviewed diff never reads them. Duplicating stream state in both `event_stream()` and `RunRecord` increases the surface area of the cancellation flow without buying behavior.
