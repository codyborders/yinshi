## Design Findings

- MINOR: `readEntryScriptSignature()` infers the app build signature by grabbing the first `script[type="module"][src]` in the document (`frontend/src/components/ChunkErrorBoundary.tsx:29-38`). That works with today's `frontend/index.html:27`, but it is a hidden coupling to DOM order rather than an explicit app-owned identifier. If another module script is ever inserted before the app entry, the reload dedupe key will silently start tracking the wrong asset, which can cause unnecessary reloads or skipped dedupe. A small explicit marker such as `data-app-entry` or an injected build id would make the invariant obvious.

## Code Quality Findings

- MINOR: The new tests only exercise the pure helpers (`frontend/src/components/__tests__/ChunkErrorBoundary.test.tsx:23-50`). They never mount the actual boundary and verify the lifecycle path in `componentDidCatch()` (`frontend/src/components/ChunkErrorBoundary.tsx:81-99`). That leaves the highest-risk behavior in this change unprotected: first chunk failure should call `window.location.reload()`, and a repeated failure should switch the UI into `reloadRecommended` mode instead of reloading forever. A React Testing Library test with a child that throws a chunk-like error would catch regressions in the real integration path.

## Simplification Findings

- MINOR: `shouldReloadForChunkError()` both decides and mutates state (`frontend/src/components/ChunkErrorBoundary.tsx:41-63`), but its name reads like a pure boolean predicate. That hidden side effect makes the recovery path harder to reason about than it needs to be. Either rename it to reflect the write (`markChunkReloadAndShouldReload`) or split the read/write responsibilities so the call site is self-explanatory.
