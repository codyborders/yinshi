## Design Findings

### MINOR — Upload size policy now leaks across layers
**Files:** `backend/src/yinshi/api/settings.py:26-34`, `backend/src/yinshi/services/pi_config.py`

The route now imports `_MAX_UPLOAD_BYTES` directly from `yinshi.services.pi_config`. The leading underscore signals “private to that module”, but the API layer now depends on it anyway.

Why this matters:
- The upload-size rule now lives in two places conceptually: route enforcement and service enforcement.
- Future refactors in `pi_config.py` can break the API layer unexpectedly.
- Junior engineers will have to know service internals to change an API policy.

Better shape:
- Move the limit into a small public configuration/helper module, or
- Expose a public `MAX_UPLOAD_BYTES` / `get_max_upload_bytes()` from the service layer.

That keeps the abstraction honest: the API layer depends on a public contract, not a private implementation detail.

## Performance Findings

### MINOR — Upload reader briefly holds large archives twice in memory
**File:** `backend/src/yinshi/api/settings.py:94-109`

`_read_upload_bytes()` collects each chunk into `chunks`, then returns `b"".join(chunks)`. For a near-limit upload, the process briefly holds both:
- the per-chunk `bytes` objects, and
- the final joined `bytes` object.

Why this matters:
- The hard limit is 50 MB, so peak memory for one request can be noticeably above 50 MB.
- Multiple concurrent uploads make the spike additive.

Safer options:
- Build into a `bytearray` and convert once at the end, or
- Use a `SpooledTemporaryFile` / temp file if the service can be taught to read from a stream.

This is not a release blocker today, but it is a real cost hidden behind an otherwise simple helper.

### MINOR — Concurrent first-login races still do unnecessary crypto work
**File:** `backend/src/yinshi/services/accounts.py:237-241`

`_create_user_record()` now handles the email race correctly, but it still generates and wraps a DEK before the `INSERT` proves this request won the race.

Why this matters:
- In the losing request, that crypto work is discarded.
- The new concurrency test makes this race much more realistic than before.

This is a small performance concern, not a correctness issue. If first-login stampedes become common, consider restructuring the insert path so expensive key wrapping only happens after ownership of the new row is certain.

## Simplification Findings

### MINOR — Auth cookie parsing is duplicated in two helpers
**File:** `backend/src/yinshi/api/auth_routes.py:107-120`

`_current_user_id()` and `_current_session_identity()` both:
- read `yinshi_session` from the cookie jar,
- return `None` when it is missing, and
- call closely related token-decoding helpers.

Why this matters:
- Cookie-name changes or token-shape changes must now be updated in two places.
- Authentication code is a bad place for small helper drift.

Simpler shape:
- Keep one helper that returns full session identity.
- Derive `user_id` from that helper when callers only need the user.

### MINOR — 413 construction is duplicated inside `_read_upload_bytes()`
**File:** `backend/src/yinshi/api/settings.py:86-105`

The same `HTTPException(status_code=413, detail="Uploaded archive exceeds the 50MB size limit")` is constructed in both the reported-size branch and the streaming-size branch.

This is small, but centralizing it into one helper/constant would reduce drift if the limit text or status ever changes.

## Changes Applied

Applied in follow-up patch:
- Exposed Pi upload size as public `MAX_UPLOAD_BYTES` and removed the API-layer dependency on a private service constant.
- Reworked `_read_upload_bytes()` to use a `bytearray`, which avoids the temporary double-buffering from `b"".join(chunks)`.
- Centralized the oversized-upload 413 into `_upload_too_large_http_exception()`.
- Consolidated auth cookie parsing so `_current_user_id()` now derives from `_current_session_identity()`.
- Deferred new-account DEK wrapping until after the winning user row is inserted, and added a concurrency assertion test to prove the expensive crypto path now runs once.
