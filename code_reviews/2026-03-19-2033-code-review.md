# Code Review

## Summary

I reviewed the GitHub App import/reconnect implementation across the backend API, GitHub service, workspace repair flow, and frontend error handling.

The main issues are behavioral, not stylistic:

1. Public GitHub repositories can be incorrectly gated behind GitHub connection.
2. Tenant repair can fail even when a safe local checkout still exists, which risks stranding user work.
3. Revoked-installation errors drop the manage URL, so the reconnect UX is missing in one of the most important failure cases.

## Design Findings

### MAJOR: Public-repo detection is implemented backwards

`resolve_github_clone_access()` uses the presence of a GitHub App installation as a proxy for "this repo is private". That is not a safe inference. Public repositories can also belong to an owner that has the app installed. In that case, the code requires a saved installation or raises `GitHubConnectRequiredError` before even attempting an anonymous clone.

Why this matters:

- The agreed plan explicitly preserves public GitHub imports without the GitHub connection gate.
- The current implementation can block public repo imports for users who never connected GitHub.
- It also causes public repos to be stored with an `installation_id`, which the plan says should remain `NULL`.

Relevant code:

- `backend/src/yinshi/services/github_app.py:402`
- `backend/src/yinshi/services/github_app.py:414`
- `backend/src/yinshi/services/github_app.py:419`

Recommended fix:

- Attempt anonymous clone first for GitHub URLs.
- Only resolve installation credentials after an auth-like clone failure.
- Persist `installation_id = NULL` when the anonymous clone succeeds.

### MAJOR: Repair prefers remote auth before local recovery, which can strand existing work

`ensure_repo_checkout_for_tenant()` resolves remote GitHub credentials before calling `_materialize_repo_checkout()`. If GitHub access is revoked or the app is temporarily unusable, `_resolve_remote_checkout()` raises immediately and the function never reaches the local-clone path, even when `repo["root_path"]` is still a valid repository on disk.

Why this matters:

- The intended behavior is to preserve existing local clones and worktrees and only fail when a remote operation is actually needed.
- The current ordering makes migrated repos dependent on GitHub auth even when a local checkout exists and could be copied safely.
- That is exactly the scenario where users are most likely to lose access to pre-feature work.

Relevant code:

- `backend/src/yinshi/services/workspace.py:113`
- `backend/src/yinshi/services/workspace.py:152`
- `backend/src/yinshi/services/workspace.py:159`

Recommended fix:

- First check whether `source_path` is a valid local repo and repair from it.
- Only resolve GitHub credentials if local recovery is impossible and a remote clone is actually required.

## Code Quality Findings

### MINOR: Revoked-installation errors lose the manage URL, so the reconnect path disappears

When GitHub returns `403` or `404` during token minting, `get_installation_token()` raises `GitHubInstallationUnusableError` without a `manage_url`. `resolve_github_clone_access()` does not catch and re-wrap that error with the saved installation metadata, so the API returns a structured error code without the actionable management link the frontend expects.

Why this matters:

- The product decision for revoked/disconnected installs was to preserve data and show a manage/reconnect path on the next remote operation.
- The implementation handles the error, but not the remediation.
- This will be most visible in the exact revoke case that the feature is supposed to smooth over.

Relevant code:

- `backend/src/yinshi/services/github_app.py:298`
- `backend/src/yinshi/services/github_app.py:419`
- `backend/src/yinshi/services/github_app.py:427`

Recommended fix:

- Catch `GitHubInstallationUnusableError` in `resolve_github_clone_access()`.
- Re-raise it with `manage_url=installation.html_url`.

## Simplification Findings

The current code overloads `installation_id is None` to mean several different states:

- public repo cloned anonymously
- GitHub repo with no app installation
- GitHub repo whose installation is not connected for this user

That makes the control flow harder to reason about and is part of why the public/private behavior regressed. A small explicit enum or tagged result type for clone-access resolution would make the code safer and easier to extend.
