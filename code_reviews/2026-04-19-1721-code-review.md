**Date:** 2026-04-19
**Branch:** `codyborders/pi-config-upload`
**Commit under review:** `61bfb43` — *"Surface imported pi skills, prompts, and extensions in slash command palette"*
**Reviewers (AI agents):** software-design skill, code-simplicity-reviewer, kieran-python-reviewer, security-sentinel, performance-oracle

## Summary

The commit does exactly what the title says: when a user imports a `pi-backup.zip`, the skills, prompts, and extension-registered slash commands from their config now show up in the chat palette (type `/` to see them). Before this change, the palette was a hardcoded list of five Yinshi-only UI commands; skills from imported configs were being loaded by pi but had no way to be discovered by the user.

**Design quality:** Solid. The change respects existing module boundaries and extends an established pattern (sidecar request/response) correctly. There are two small-but-real design smells: a helper function that doesn't earn its existence (`mapLoaderArray`) and a set of three Pydantic models that could be one with a `kind` discriminator.

**Security:** No critical vulnerabilities. `agent_dir` is server-derived from authenticated tenant context — an attacker cannot steer it to read arbitrary host paths. React's built-in escaping mitigates the obvious XSS vector in rendering skill descriptions. However, one **HIGH** severity DoS vector exists: the endpoint re-evaluates every extension's JavaScript on every call, and no cache is in place. A user with a malicious or expensive extension can stall the shared Node sidecar event loop every time any session page is opened.

**Performance:** The biggest real-world problem. `GET /api/settings/pi-config/commands` is called on every Session component mount with no client-side cache, and the sidecar reloads the entire pi resource tree (scanning the filesystem and evaluating every extension module) on every call. For a tenant with 10 extensions opening 3 sessions, that's ~30 extension-module evaluations per session-navigation session. Two small changes (cache on sidecar keyed by mtime; hoist the hook state into an app-level context or a module-level promise) reduce this by ~100x.

**Code quality:** Python files score OK against Kieran's bar but miss a few items: zero assertions in the new route, a `finally` block that can leak sockets if one cleanup call throws, and a too-generic method name (`list_resources`). Tests were added for the frontend path (via mock in `Session.test.tsx`) but no backend test covers the new route.

**Simplification opportunities:** Several. `refreshKey` parameter in `usePiCommands` is unused dead weight. `isMountedRef` is pre-React-18 defensive code. `mapLoaderArray` helper is called twice with no real compression. A 404-branch guard for a status code the backend never returns. Return values (`loading`, `error`) that no caller reads. ~60-80 lines could come out without changing behavior.

**Top-priority fixes (in order):**
1. **[SECURITY + PERF]** Cache `list_resources` on the sidecar keyed by `(agent_dir, mtime)`. Fixes the extension re-evaluation DoS and the biggest perf hit in one change.
2. **[PERF]** Hoist `usePiCommands` state into an app-level provider or use a module-level promise cache. One fetch per tenant session instead of one per Session page mount.
3. **[SECURITY]** Drop `extension_path` from the response. Frontend never uses it, it leaks host directory layout.
4. **[CODE QUALITY]** Add backend tests for the new route: short-circuit path (no agent_dir), success path (mock sidecar returns), error path (503).
5. **[CODE QUALITY]** Guard the `finally` block cleanup calls individually so a failing `end_tenant_container_activity()` doesn't leak the sidecar socket.

Only if you have time after those: collapse the three Pydantic models into one, delete dead simplification candidates.

## Design Findings

*(From "A Philosophy of Software Design" — shallow modules, information leakage, complexity red flags, interface precision.)*

### D1. `mapLoaderArray` is a shallow abstraction — inline or remove

**What**: In `sidecar/src/sidecar.js:246-254`, a helper called `mapLoaderArray(loaderResult, property, transform)` wraps a `Array.isArray(...).map(...)` dance. It's used exactly twice (for `skills` and `prompts`). The third category — `extensions` — does *not* use this helper because `getExtensions()` returns a different shape.

**How the code works today**:
```js
function mapLoaderArray(loaderResult, property, transform) {
  const values = loaderResult?.[property];
  if (!Array.isArray(values)) {
    return [];
  }
  return values.map(transform);
}

// ...
skills: mapLoaderArray(loader.getSkills(), "skills", toSkillSummary),
prompts: mapLoaderArray(loader.getPrompts(), "prompts", toPromptSummary),
extension_commands: extensions.flatMap(toExtensionCommandSummaries),  // bypasses the helper
```

**Why it's a problem**: A good helper compresses a repeated idiom into a simpler interface. This helper does the opposite — it takes three arguments (one of which is a string property name, which loses type safety in TypeScript and editor autocomplete) to do something that is two lines of inline code. Worse, the asymmetric third case breaks the symmetry the helper was meant to provide. A reader sees "`mapLoaderArray`" and has to look up what it does, verify the string property name matches a real field, then ask "why doesn't `extensions` use this?" — net cognitive cost is *higher*, not lower.

**What to do**: Inline it.
```js
const skillsResult = loader.getSkills();
const promptsResult = loader.getPrompts();
return {
  skills: (skillsResult?.skills ?? []).map(toSkillSummary),
  prompts: (promptsResult?.prompts ?? []).map(toPromptSummary),
  extension_commands: extensions.flatMap(toExtensionCommandSummaries),
};
```
Now all three lines look parallel, the shape of `getSkills()`'s return is visible, and there's no helper to look up.

### D2. `_EMPTY_PI_COMMANDS` doing two jobs at once

**What**: `backend/src/yinshi/api/settings.py:218-222` defines a module-level dict used as both the 404-fallback *value* and as the key *schema* for projecting the sidecar response.

**How**:
```python
_EMPTY_PI_COMMANDS: dict[str, list[Any]] = {
    "skills": [],
    "prompts": [],
    "extension_commands": [],
}

# Used as a fallback value:
if tenant_sidecar_context.agent_dir is None:
    return dict(_EMPTY_PI_COMMANDS)

# And used as a key source:
return {key: resources.get(key, []) for key in _EMPTY_PI_COMMANDS}
```

**Why it's a problem**: When one object holds two distinct responsibilities, changing one ripples into the other. If a future engineer adds a new command category (e.g. `"agents"`), they must remember both that it changes the default payload *and* that it defines the projection shape. A single change (add a key) now has two logically independent consequences, and the relationship isn't documented.

**What to do**: Split the two concerns.
```python
_PI_COMMAND_KEYS = ("skills", "prompts", "extension_commands")

# Fallback value:
if tenant_sidecar_context.agent_dir is None:
    return {key: [] for key in _PI_COMMAND_KEYS}

# Projection:
return {key: resources.get(key, []) for key in _PI_COMMAND_KEYS}
```

### D3. Three Pydantic models where one would do

**What**: `backend/src/yinshi/models.py:413-446` defines `PiSkillCommand`, `PiPromptCommand`, `PiExtensionCommand`, and `PiConfigCommandsOut` (the wrapper). The three command models share three fields (`name`, `description`, `command_name`) and differ by one field each (`disable_model_invocation` on skill, `extension_path` on extension, nothing on prompt).

**Why it matters**: The three types carry information that could be useful (knowing whether a palette entry is a skill vs an extension lets you style them differently, group them, etc.). But `frontend/src/pages/Session.tsx:327-339` immediately flattens them into a single `SlashCommand[]`, losing the distinction. That's 6 schemas (3 Python + 3 TypeScript) maintained for a distinction that exists nowhere in the rendered UI.

**Two viable designs** (per *A Philosophy of Software Design*, Ch. 16 — "Design it twice"):

**Option A** — collapse to one model with a discriminator:
```python
class PiCommand(BaseModel):
    kind: Literal["skill", "prompt", "extension"]
    name: str
    description: str = ""
    command_name: str
    disable_model_invocation: bool = False
    extension_path: str | None = None

class PiConfigCommandsOut(BaseModel):
    commands: list[PiCommand] = Field(default_factory=list)
```

**Option B** — keep three models but extract a shared base:
```python
class _PiCommandBase(BaseModel):
    name: str
    description: str = ""
    command_name: str

class PiSkillCommand(_PiCommandBase):
    disable_model_invocation: bool = False

class PiPromptCommand(_PiCommandBase):
    pass

class PiExtensionCommand(_PiCommandBase):
    extension_path: str | None = None
```

**Recommendation**: Option A. It's simpler, matches what the frontend actually needs (a flat palette), and preserves the `kind` distinction for future grouping. Option B keeps three schemas with a marker class (`PiPromptCommand` with no extra fields) — a code smell itself.

### D4. `YINSHI_COMMAND_NAMES` set collision is possible

**What**: `frontend/src/components/ChatView.tsx:16, 79-101`. A `Set` of Yinshi's UI command names (`help`, `model`, `tree`, `export`, `clear`) is used to decide whether to dispatch immediately (Yinshi command) or insert into the input for arg editing (pi command).

**How**:
```js
const YINSHI_COMMAND_NAMES = new Set(SLASH_COMMANDS.map((c) => c.name));

// In selectCommand:
if (YINSHI_COMMAND_NAMES.has(name)) {
  // ... dispatch as Yinshi command
} else {
  // ... insert as pi command
}
```

**Why it's fragile**: Skill names come from user zips. A user who names a skill `help`, `tree`, etc. will see their skill silently collide with the builtin — clicking their skill in the palette will trigger the Yinshi `/help` handler. Pi skills have a naming convention (`/skill:<name>`), so today this is mostly unreachable (their command_name is `skill:caveman`, not `caveman`). But prompt templates in pi use just the bare name, so a user-authored prompt named `help` *will* collide.

**What to do**: Prefer type-driven discrimination over name matching. Either:
- Tag `SlashCommand` with `source: "builtin" | "pi"` so classification is carried in the data.
- Namespace Yinshi commands (e.g. always register them as `yinshi:help`).

### D5. Flatten-and-map in the wrong place

**What**: `frontend/src/pages/Session.tsx:327-339` converts three typed arrays (`skills`, `prompts`, `extension_commands`) into a single `SlashCommand[]` via a `useMemo`.

**Why**: `Session` is a page component — its job is routing/layout/state for a session view. Mapping DTOs to view models is an adapter concern that shouldn't live in a page. A reader of `Session.tsx` should see "here's what the page does"; instead they see data-shape gymnastics.

**What to do**: Push `piSlashCommands` into `usePiCommands` (return both the raw structured payload *and* the flat `SlashCommand[]`), or create a sibling selector hook `usePiSlashPalette()`. The page body then reads as `const palette = usePiSlashPalette()` — one line of intent.

### D6. `list_resources` method name is too generic

**What**: `backend/src/yinshi/services/sidecar.py:285`. Both the sidecar-client method and the wire-protocol type are named `list_resources` / `"list_resources"`.

**Why**: "Resources" is vague — it could mean skills, prompts, extensions, themes, models, credentials, sessions, anything. The actual domain concept is "slash commands discoverable from an imported pi config." A name like `list_imported_commands` or `list_pi_commands` matches what the caller *thinks about*, not what the JSON envelope happens to be called.

**What to do**: Rename the Python method to `list_imported_commands`. The wire-protocol type string is fine as-is — it's internal.

### D7. Socket call dance is repeated in two routes

**What**: `backend/src/yinshi/api/settings.py:237-253` (new) and `backend/src/yinshi/api/catalog.py:65-79` (existing). Both follow the identical pattern:
```
begin_tenant_container_activity(...)
try:
    sidecar = await create_sidecar_connection(...)
    result = await sidecar.<operation>(...)
except (OSError, SidecarError):
    raise HTTPException(503, ...)
finally:
    end_tenant_container_activity(...)
    touch_tenant_container(...)
    if sidecar is not None:
        await sidecar.disconnect()
```

**Why it matters**: Two identical shapes = the first call for a helper. A third route appearing with the same pattern would confirm it.

**What to do (future commit)**: Extract an async context manager in `sidecar_runtime.py`:
```python
async with tenant_sidecar_call(request, tenant) as sidecar:
    result = await sidecar.<operation>(...)
return result
```
Not required for this commit, but flag it: when the third call site appears, do the refactor.

## Security Findings

### S1. [HIGH] DoS via repeated extension code execution on every session open

**Files**: `sidecar/src/sidecar.js:266-291`, `backend/src/yinshi/api/settings.py:225-258`, `frontend/src/hooks/usePiCommands.ts:60-72`

**What**: Every time a user opens a Session page, the frontend hook calls `GET /api/settings/pi-config/commands`. The backend opens a fresh sidecar connection. The sidecar constructs a new `DefaultResourceLoader` and calls `loader.reload()`. Inside `reload()`, pi uses `jiti.import(extensionPath, {default: true})` with `moduleCache: false` (`@mariozechner/pi-coding-agent/dist/core/extensions/loader.js:231`) to evaluate every extension's JavaScript. Because caching is disabled, every palette fetch pays the full parse + compile + execute cost for every extension.

**Why this is a problem**:
- A user can upload a `pi-backup.zip` containing an extension file like `while(true){}`. That's their own tenant, so the blast radius depends on your deploy mode:
  - **Container mode ON** (default): each tenant has their own sidecar container. The attacker freezes *their own* container. Limited impact, but still a UX hit for them.
  - **Container mode OFF**: one shared sidecar serves all tenants. A single tenant's extension wedges the shared Node event loop. All other users experience stream jitter or complete stalls.
- Even *benign* extensions pay the full jiti cost every call. With 10 small extensions averaging 15ms cold-load each, you spend ~150ms on `await loader.reload()` *before* the endpoint can even read skills/prompts.
- The frontend has no cache: opening three sessions = three full reloads = three rounds of extension execution.

**How to fix** (pick one, ranked by effort):
1. **Cheapest + highest ROI**: Cache the resource payload on the sidecar keyed by `(agent_dir, mtime-of-agent-dir-tree)`. When the pi-config importer writes new files (a discrete event that already exists in `pi_config.py`), invalidate the cache. Post-fix, the steady-state cost is a single `fs.stat` walk, not a full reload.
2. Simpler fallback: 30-second TTL cache on the sidecar.
3. Add a rate limit to the route (`@limiter.limit("20/minute")` on the FastAPI route) so a single tenant can't DoS themselves by rapid session navigation.

### S2. [MEDIUM] Host-path information disclosure via `extension_path`

**Files**: `sidecar/src/sidecar.js:231` (in `toExtensionCommandSummaries`), `backend/src/yinshi/models.py:430-435` (`PiExtensionCommand.extension_path`)

**What**: The `extension_path` field returned to the client is the raw absolute filesystem path where pi loaded the extension from. That path includes the host's directory layout — `/data/.pi-config/agent/extensions/foo.ts` in container mode, or the real host path like `/home/user/data/tenants/<user_id>/.pi-config/agent/extensions/foo.ts` if container mode is disabled. The frontend never uses this field.

**Why it matters**: It's authenticated-user-only (a user can only see their own paths), so it's not a cross-tenant leak. But:
- In non-container mode, it leaks the host's tenant data directory layout, which can include sensitive identifiers like `user_id`.
- It's gratuitous — no client code reads `extension_path`. Minimum exposure principle says don't ship data nobody needs.

**How to fix**: Either drop the field entirely from `toExtensionCommandSummaries`, or return only `path.basename(extensionPath)` (just the filename).

### S3. [MEDIUM] Stored input rendering fragility in skill/prompt descriptions

**Files**: `frontend/src/components/SlashCommandMenu.tsx:48`, `frontend/src/components/ChatView.tsx:246` (indirect)

**What**: Skill/prompt descriptions come from user-uploaded markdown frontmatter — pure user input, not validated. They're rendered in the palette as `<span>{cmd.description}</span>`.

**Why it's currently safe, but brittle**: React escapes text children, so raw script content in a description *cannot* execute today. What's fragile:
- No length cap. A skill with a 1 MB description ships to the browser on every palette render. Performance hit, not a security hole — but worth capping.
- Skill `name` is concatenated into `command_name = \`skill:${skill.name}\`` (`sidecar.js` `toSkillSummary`) without validation. A name containing newlines, null bytes, or control characters would render weirdly in the palette. If any future code path switches to `dangerouslySetInnerHTML` or feeds these strings to a URL, the safety evaporates silently.

**How to fix**: Validate skill/prompt names against `^[a-zA-Z0-9_:.-]{1,64}$` when building summaries in the sidecar, and truncate description to, say, 240 chars:
```js
function safeName(name) {
  if (typeof name !== "string" || !/^[a-zA-Z0-9_:.-]{1,64}$/.test(name)) return null;
  return name;
}
function safeDescription(description) {
  if (typeof description !== "string") return "";
  return description.length > 240 ? description.slice(0, 240) : description;
}
```
Skip entries with invalid names. This also defends against S4 below.

### S4. [LOW] No length or count bounds on the response

**Files**: `sidecar/src/sidecar.js:266-291`

**What**: No limit on how many skills/prompts/extensions the endpoint returns, nor on per-field sizes.

**Why it matters**: An attacker's own zip with 100,000 synthetic skill files would balloon the JSON-line response (which Python reads as a *single line* via `_read_line`). Memory pressure, response time, browser rendering — all hurt.

**How to fix**: Cap each category at, say, 500 entries. If a real user exceeds that, it's almost certainly a misconfiguration worth surfacing as a warning.

### S5. Not-findings (verified safe)

For the record, the security reviewer verified these potential concerns as *not* real:

- **`agentDir` injection**: The path is derived entirely server-side from `tenant.data_dir` via `resolve_effective_pi_runtime(tenant.user_id, tenant.data_dir)` and `_resolve_agent_dir_for_runtime` which asserts `is_path_inside(..., data_dir)` before remap. An attacker cannot steer it to read arbitrary host paths.
- **Tenant isolation**: `require_tenant(request)` gates the endpoint; `data_dir` is scoped to the authenticated `user_id`.
- **Slash command submit traversal**: A malicious skill name like `skill:../../etc/passwd` is sent as text through `session.prompt()`. Pi resolves skills by registered name, not filesystem lookup — the traversal string is just inert text.
- **Error-path leakage**: The 503 body is the static constant `"Agent environment temporarily unavailable"`. Internal errors stay in backend logs.
- **XSS**: All rendering uses `{cmd.description}` / `{cmd.name}` text-node interpolation. React auto-escapes.

## Performance Findings

### P1. [HIGH] Extension JS evaluated on every palette fetch (no cache)

*Same issue as S1 but framed as perf.*

**File**: `sidecar/src/sidecar.js:287-300`

**What happens every call**:
1. Fresh Unix socket connect + init handshake.
2. `new DefaultResourceLoader({ cwd, agentDir })`.
3. `await loader.reload()` — which recursively scans `agentDir` for skills, prompts, extensions, themes, and AGENTS.md files.
4. For every extension file found, pi runs `jiti.import(...)` which does synchronous parse + compile + module evaluation.
5. Returns the list, closes the socket.

**Projected cost**: 10 extensions × 15 ms cold-load ≈ 150 ms per request, *before* reading skills/prompts. Scan cost scales linearly with `agentDir` size (typical pi configs have hundreds of files).

**Mitigation**: Cache on sidecar keyed by `(agent_dir, mtime)`. Invalidate on pi-config import events (which already exist in `pi_config.py`).

### P2. [HIGH] No client-side caching across Session mounts

**Files**: `frontend/src/hooks/usePiCommands.ts:60-67`, `frontend/src/pages/Session.tsx:28`

**What**: The hook fires a fetch from `useEffect` on every Session component mount. Opening three sessions in a row triggers three identical requests.

**Why it matters**: The data is tenant-wide and identical across sessions. There's no React Query / SWR, no module-level cache, no context. Compounds directly with P1 — three fetches = three full extension re-evaluations on the sidecar.

**Mitigation**: Either:
- Hoist to an app-level context (`PiCommandsProvider` at the root), fetch once per tenant session.
- Use a module-level promise cache in `usePiCommands.ts`:
  ```ts
  let pendingFetch: Promise<PiConfigCommands> | null = null;
  let cached: PiConfigCommands | null = null;
  // ... use the cached value on mount if present
  ```
Revalidate only on explicit invalidation events (pi-config import, category toggle).

### P3. [MEDIUM] Themes and AGENTS.md loaded then discarded

**File**: `sidecar/src/sidecar.js:303-310`

**What**: `loader.reload()` loads skills + prompts + extensions + **themes** + **agent-files**. The endpoint only reads skills/prompts/extensions. Themes and agents-files are I/O-discarded.

**Why it matters**: Each is an additional directory scan (and, for agents-files, an ancestor-directory walk from `cwd` up to `/`). On a warm filesystem it's cheap, but it's pure waste.

**Mitigation**: If pi's SDK exposes per-category loaders (`loadSkills`, `loadPrompts`, `loadExtensions`), use those directly instead of `reload()`. Cuts two extra directory scans per call.

### P4. [MEDIUM] Fresh Unix-socket connect/disconnect per request

**File**: `backend/src/yinshi/api/settings.py:245-256`

**What**: Each call to `/pi-config/commands` opens a new sidecar connection, reads `init_status`, makes one request, disconnects.

**Why it matters (low priority, but stacks)**: Per-request socket cost is small (~1-2 ms locally), but it's fixed overhead on every call. Once P2 fixes the client-side duplication, this becomes the remaining floor.

**Mitigation**: Pool sidecar connections per tenant for lightweight read-only operations. Not worth doing until P1/P2 are in place.

### P5. [MEDIUM] Batch with existing catalog endpoint

**Files**: `frontend/src/hooks/useCatalog.ts`, `frontend/src/hooks/usePiCommands.ts`, `backend/src/yinshi/api/catalog.py`, `backend/src/yinshi/api/settings.py`

**What**: `Session.tsx` mounts `useCatalog` and `usePiCommands` in parallel. Both hit the sidecar. Catalog and commands come from the same loader infrastructure.

**Mitigation**: Combine into one `/api/settings/pi-config/bootstrap` endpoint returning both. One sidecar round trip instead of two. Halves connection overhead and network RTT (p50 ~200ms → ~100ms).

### P6. Sidecar event-loop concern

**Flagged by the reviewer as a correction to an assumption I made**: I assumed that using a fresh socket connection for `list_resources` isolated it from concurrent `query` operations. That's true at the *socket-demux* level (they're on different TCP connections and don't interleave messages). It is **false at the event-loop level** — `jiti.import(...)` does a synchronous compile + run inside the Node event loop. A concurrent streaming `query` on a different socket will stutter for the duration of extension evaluation.

**What to do**: Part of the P1 cache-everything fix. Additionally, if cache misses are frequent, move extension evaluation into a `worker_threads` worker with a timeout.

## Code Quality Findings

*(From Kieran Python reviewer, applied to the three `.py` files in the diff.)*

### Q1. Zero assertions at the new route boundary

**File**: `backend/src/yinshi/api/settings.py:227` (`list_pi_config_commands`)

**What**: The project's style guide (CLAUDE.md + PYTHON.md) requires a minimum of two assertions/defensive checks per function, especially at boundaries. The new route has none.

**Why it matters**: Boundaries (API entry points) are where bad input is most likely and cheapest to catch. An assertion is also a piece of executable documentation.

**Where they belong**:
```python
@router.get("/pi-config/commands", response_model=PiConfigCommandsOut)
async def list_pi_config_commands(request: Request) -> dict[str, Any]:
    tenant = require_tenant(request)
    assert tenant is not None, "require_tenant returned without raising"  # programmer error

    try:
        tenant_sidecar_context = await resolve_tenant_sidecar_context(request, tenant)
    except (ContainerStartError, ContainerNotReadyError) as error:
        raise HTTPException(status_code=503, detail=_SIDECAR_UNAVAILABLE_DETAIL) from error

    if tenant_sidecar_context.agent_dir is None:
        return {key: [] for key in _PI_COMMAND_KEYS}

    # positive-space assertion before opening the socket:
    assert tenant_sidecar_context.socket_path or True  # socket_path may be None in non-container mode

    sidecar = None
    # ...
    resources = await sidecar.list_resources(agent_dir=tenant_sidecar_context.agent_dir)
    assert isinstance(resources, dict), "sidecar list_resources contract: returns dict"  # negative-space
```

### Q2. `finally` block can leak sockets

**File**: `backend/src/yinshi/api/settings.py:248-252`

**What**: The `finally` block runs three cleanup steps in sequence with no guards:
```python
finally:
    end_tenant_container_activity(request, tenant)
    touch_tenant_container(request, tenant)
    if sidecar is not None:
        await sidecar.disconnect()
```

**Why this is the "92% of catastrophic failures" pattern**: If `end_tenant_container_activity` raises (say, the container manager is in a bad state), `touch_tenant_container` and `sidecar.disconnect()` never run. The socket leaks. Over time you accumulate dangling sidecar connections.

**How to fix**: Guard each cleanup call.
```python
finally:
    try:
        end_tenant_container_activity(request, tenant)
    except Exception:
        logger.exception("end_tenant_container_activity failed")
    try:
        touch_tenant_container(request, tenant)
    except Exception:
        logger.exception("touch_tenant_container failed")
    if sidecar is not None:
        try:
            await sidecar.disconnect()
        except Exception:
            logger.exception("sidecar disconnect failed")
```
Or extract to a helper / async context manager so every call site gets the same safety.

### Q3. Narrow `except` may miss real errors

**File**: `backend/src/yinshi/api/settings.py:241`

**What**: `except (OSError, SidecarError)` is the catch. But `_read_line` in `sidecar.py:114` calls `json.loads(line.decode())` — a malformed response raises `json.JSONDecodeError` (subclass of `ValueError`), which slips through. Async operations can also raise `asyncio.TimeoutError`.

**How to fix**:
```python
except (OSError, SidecarError, json.JSONDecodeError, asyncio.TimeoutError) as error:
    logger.warning("sidecar list_resources failed", exc_info=True)
    raise HTTPException(status_code=503, detail=_SIDECAR_UNAVAILABLE_DETAIL) from error
```

### Q4. No logging on 503 paths

**File**: `backend/src/yinshi/api/settings.py:229, 241-244`

**What**: When the sidecar is unavailable, the user sees a generic 503. The backend logs nothing. Operators will have no trail when investigating why users saw transient errors.

**How to fix**: Add `logger.warning("pi-config/commands 503: ...", exc_info=True)` on both except branches (container-unavailable and sidecar-unavailable). The style guide's "log and raise" pattern.

### Q5. Hardcoded `request_id` can collide

**File**: `backend/src/yinshi/services/sidecar.py:287`

**What**: `request_id = "list_resources"`. The same string is used for every call. Other methods use per-call unique IDs (`f"resolve-{model_key}"`).

**Why it matters**: If two `list_resources` requests are ever in flight on the same connection (not today's architecture — each request opens a fresh connection — but the invariant isn't documented), responses could be mismatched.

**How to fix**: Either use `uuid4().hex`, or document the single-request-per-connection invariant with a comment above `_send`.

### Q6. Three copies of send → read → validate-type

**File**: `backend/src/yinshi/services/sidecar.py` — `resolve_model` (L221), `get_catalog` (L266), `list_resources` (L285)

**What**: All three methods follow the same shape. The new method makes it three-of-a-kind; the style guide's tolerance for duplication is "two is fine, three signals extraction."

**Proposed helper**:
```python
async def _request_typed(
    self,
    payload: dict[str, Any],
    *,
    expected_type: str,
    op_label: str,
) -> dict[str, Any]:
    """Send a typed request and return the matching reply, or raise SidecarError."""
    await self._send(payload)
    msg = await self._read_line()
    if msg is None:
        raise SidecarError(f"Sidecar connection lost during {op_label}")
    if msg.get("type") == "error":
        raise SidecarError(f"{op_label} failed: {msg.get('error', 'unknown')}")
    if msg.get("type") != expected_type:
        raise SidecarError(f"{op_label} unexpected response type: {msg.get('type')}")
    return msg
```
Each public method shrinks to ~5 lines. Not required for this commit — flag for follow-up.

### Q7. `list_resources` return type is vague

**File**: `backend/src/yinshi/services/sidecar.py:285`

**What**: Returns `dict[str, Any]` including wire-envelope keys (`id`, `type`) that the caller has to remember to strip.

**How to fix**: Use a `TypedDict` and strip the envelope here:
```python
class PiResourcesPayload(TypedDict):
    skills: list[dict[str, Any]]
    prompts: list[dict[str, Any]]
    extension_commands: list[dict[str, Any]]

async def list_imported_commands(self, *, agent_dir: str | None = None) -> PiResourcesPayload:
    msg = await self._request_typed(
        {"type": "list_resources", "id": "list_resources",
         "options": {"agentDir": agent_dir} if agent_dir else {}},
        expected_type="resources",
        op_label="list_resources",
    )
    return {
        "skills": msg.get("skills", []),
        "prompts": msg.get("prompts", []),
        "extension_commands": msg.get("extension_commands", []),
    }
```
