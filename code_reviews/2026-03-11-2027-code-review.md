# Yinshi Full Codebase Review -- 2026-03-11

## Summary

This is a comprehensive review of the entire Yinshi codebase: a FastAPI backend, Next.js frontend, and Node.js sidecar application for managing coding agent workspaces. The review was conducted across five dimensions: architectural design, security, performance, Python code quality, and simplification opportunities.

**Scope:** 20 backend Python files, 1 sidecar (Node.js), ~15 frontend (TypeScript/React) files.

**Key Takeaways:**
- **Security risk is HIGH.** A WebSocket header bypass allows unauthenticated access to any endpoint. The `validate_user_path` function exists but is never called in production, leaving path traversal wide open. The `DISABLE_AUTH` flag fails open instead of closed.
- **Performance ceiling is ~10-20 concurrent streaming users** due to synchronous SQLite calls blocking the async event loop, connection-per-call churn, and sidecar sessions that never get evicted from memory.
- **Significant code duplication** across schema definitions, owner-check functions, PATCH handlers, and frontend components. An entire `Dashboard -> RepoList -> WorkspaceView` component chain appears to be dead code.
- **No changes were applied** in this review. All findings are documented for prioritized action.

---

## Design Findings

These findings come from applying the principles in "A Philosophy of Software Design" (Ousterhout) to the codebase.

### D1. Duplicated Schema SQL Between `db.py` and `tenant.py` (Information Leakage)

**What:** The database table definitions (repos, workspaces, sessions, messages) are defined twice -- once in `backend/src/yinshi/db.py` (lines 16-73, the legacy single-DB schema) and again in `backend/src/yinshi/tenant.py` (lines 40-96, the per-user schema). The only difference is that the legacy schema includes an `owner_email` column on the `repos` table.

**How it works:** When a new user signs up, `init_user_db()` in `tenant.py` creates their personal SQLite database using `USER_SCHEMA_SQL`. Meanwhile, `db.py` has its own `SCHEMA_SQL` for the legacy single-database mode. Both define the same four tables with the same columns, indexes, and triggers -- about 50 lines of SQL duplicated.

**Why this is a problem:** This is a textbook case of *information leakage* -- the same design decision (the database schema) is encoded in two separate modules. If someone adds a column to the sessions table in `tenant.py` but forgets to update `db.py`, the legacy migration code will silently produce a different schema. More practically, any schema change requires editing two files and keeping them in sync by hand, which is error-prone.

**Fix:** Extract a shared base schema (e.g., `_BASE_TABLE_SQL`) and have both `SCHEMA_SQL` and `USER_SCHEMA_SQL` compose it with their specific additions.

---

### D2. Duplicated `_check_workspace_owner` Across 3 Files (Repetition)

**What:** The function `_check_workspace_owner` is defined identically in `api/workspaces.py` (lines 29-39) and `api/sessions.py` (lines 15-25). A third variant exists in `api/stream.py` (lines 86-89) that checks ownership differently (using a pre-joined query result).

**How it works:** Each function takes a database connection, workspace ID, and user email. It runs a 3-table JOIN (sessions -> workspaces -> repos) to verify the caller owns the workspace. If not, it raises a 404.

**Why this is a problem:** Identical code in multiple places means a bug fix must be applied everywhere. The `stream.py` variant already diverged in implementation, proving that copies drift over time. This is the "Repetition" red flag.

**Fix:** Move the canonical implementation to `api/deps.py` alongside the existing `check_owner` helper. Import it where needed.

---

### D3. Sidecar Protocol Knowledge Leaked Across Python/Node Boundaries (Information Leakage)

**What:** The sidecar communication protocol (JSON message types, field names, event structures) is hardcoded in both `backend/src/yinshi/services/sidecar.py` (the Python client) and `sidecar/src/sidecar.js` (the Node.js server). Neither side references a shared schema.

**How it works:** The Python client constructs messages like `{"type": "query", "id": sessionId, "prompt": prompt, "options": {...}}` and parses responses looking for fields like `data.type == "result"`, `data.usage`, `data.provider`. The sidecar constructs these same messages independently. If a field is renamed or a new message type is added on one side, the other side breaks silently.

**Why this is a problem:** The protocol is a shared design decision that lives in two places across two languages. Changes require coordinated edits to both files with no compile-time or schema validation catching mismatches.

**Fix:** Define the protocol in a shared JSON Schema or at minimum a `PROTOCOL.md` document. Consider adding protocol version negotiation to the init handshake.

---

### D4. Temporal Decomposition in `_migrate_legacy_data` (Temporal Decomposition)

**What:** The legacy data migration in `services/accounts.py` (lines 40-115) processes data in strict temporal order: first repos, then workspaces, then sessions, then messages. Each level iterates independently with nested loops.

**How it works:** The function opens the legacy DB, reads all repos for the user's email, then for each repo reads all workspaces, then for each workspace reads all sessions, then for each session reads all messages. Each entity is individually inserted into the new per-user DB.

**Why this is a problem:** This creates an O(repos * workspaces * sessions * messages) loop that loads all data through Python objects. SQLite's `ATTACH DATABASE` + `INSERT INTO ... SELECT FROM` would be far more efficient and eliminate the nested loops entirely.

---

### D5. Special-General Mixture in `get_db_for_request` (Special-General Mixture)

**What:** The `get_db_for_request` function in `api/deps.py` (lines 14-23) serves dual duty: it returns a per-user database in multi-tenant mode, or the legacy shared database when no tenant is present.

**How it works:**
```python
def get_db_for_request(request: Request):
    tenant = get_tenant(request)
    if tenant:
        return get_user_db(tenant)
    return get_db()
```

**Why this is a problem:** Every call site that uses this function must implicitly handle both modes. The caller cannot know which database it received. The two modes have different schemas (legacy has `owner_email`, tenant does not), so queries that reference `owner_email` only work in one mode. This is the "special-general mixture" red flag -- a general-purpose mechanism contaminated by special-case logic.

---

### D6. `_summarize_prompt` Embedded in `stream.py` (Different Layer, Different Abstraction)

**What:** The `_summarize_prompt` function (stream.py lines 40-83) is an NLP-style text processing function that strips filler phrases and extracts keywords from user prompts to generate workspace names. It lives inside the SSE streaming module.

**How it works:** It iterates over 13 filler prefixes, strips them, removes stop words, and returns the first 5 remaining words joined by spaces. This is purely a text utility with no connection to streaming.

**Why this is a problem:** The streaming module should deal with SSE events and sidecar communication, not text processing. This function belongs in `services/workspace.py` or a utility module, keeping each module at a single level of abstraction.

---

### D7. Private Function `_make_tenant` Imported Cross-Module (Nonobvious Code)

**What:** The function `_make_tenant` in `tenant.py` has a leading underscore (Python convention for "private"), but it is imported and used by `auth.py` and `services/accounts.py`.

**Why this is a problem:** The underscore convention signals "do not use outside this module." When other modules import it anyway, readers are confused about whether it is truly private or just poorly named. Either remove the underscore (making it part of the public API) or restructure so external callers use a public function instead.

## Security Findings

These findings come from the security-sentinel agent. Severity levels: CRITICAL (exploit now), HIGH (significant risk), MEDIUM (defense-in-depth), LOW (hardening).

### SEC-C1. [CRITICAL] WebSocket Header Bypass -- Complete Auth Bypass

**What:** In `auth.py` (lines 103-105), the authentication middleware explicitly skips authentication for any request with an `Upgrade: websocket` header:

```python
if request.headers.get("upgrade", "").lower() == "websocket":
    return await call_next(request)
```

**How it works:** An attacker can send a regular HTTP POST/GET to any protected endpoint (e.g., `/api/sessions/123/prompt`) and add the header `Upgrade: websocket`. The middleware sees this header and skips all authentication. The WebSocket upgrade never actually completes (there are no WebSocket handlers), but the request has already been passed through to the route handler without any identity check.

**Why this is dangerous:** This is a complete authentication bypass. Any endpoint in the application can be accessed without a session cookie by any network-reachable client. The comment says "auth checked in WS handler itself" but no WebSocket handlers exist.

**Fix:** Remove the WebSocket bypass entirely. There are no WebSocket endpoints in the codebase.

---

### SEC-C2. [CRITICAL] `validate_user_path` Defined But Never Called -- Path Traversal

**What:** The function `validate_user_path()` in `tenant.py` (line 28) validates that a filesystem path stays within a user's data directory. However, it is **never called from any production code**. It only appears in test files.

**How it works:** Without this validation:
- The `import_repo` endpoint (repos.py line 64) allows a tenant to register a `local_path` pointing anywhere on the filesystem.
- The `get_session_tree` endpoint (sessions.py line 136) calls `os.walk()` on a `workspace_path` from the database with no path validation.
- The sidecar receives `cwd` paths from the backend (stream.py line 197) and the coding agent can read/write files and run shell commands within that directory.

**Why this is dangerous:** A malicious user could set `cwd` to `/etc`, `/root/.ssh`, or another user's data directory. The coding agent would then operate with full filesystem access at that location.

**Fix:** Call `validate_user_path()` in `import_repo` (for `local_path`), in workspace creation (for worktree path), and in the stream endpoint before passing `workspace_path` to the sidecar.

---

### SEC-C3. [CRITICAL] `DISABLE_AUTH` Fails Open Instead of Closed

**What:** In `auth.py` (lines 70-75), authentication is silently disabled in two situations: when `DISABLE_AUTH=true` is set, OR when no OAuth provider is configured (both `GOOGLE_CLIENT_ID` and `GITHUB_CLIENT_ID` are empty).

**How it works:** The `_auth_disabled()` method returns `True` if no OAuth credentials are configured. When auth is disabled, the middleware returns early without setting `request.state.tenant`, so every `get_tenant()` call returns `None` and every `check_owner()` call succeeds.

**Why this is dangerous:** A production misconfiguration where OAuth environment variables are accidentally empty would silently disable all authentication and authorization. The application starts normally and appears to work, but every endpoint is globally accessible.

**Fix:** Decouple "no OAuth configured" from "auth disabled." If no provider is configured in production (`DEBUG=false`), the application should refuse to start, not silently disable auth.

---

### SEC-H1. [HIGH] Sidecar Unix Socket Has No Authentication

**What:** The sidecar listens on `/tmp/yinshi-sidecar.sock` with world-readable permissions. Any local user on the system can connect.

**How it works:** The sidecar accepts any connection on the Unix socket and processes commands without any authentication token or credential check. A local attacker could:
1. Send arbitrary prompts to the coding agent
2. Pass arbitrary `cwd` paths, making the agent operate on any filesystem location
3. Pass arbitrary API keys
4. Execute file reads, writes, and shell commands via the agent's tool set

**Fix:** Set socket permissions to `0700`. Move the socket out of `/tmp`. Add a shared secret token to the protocol.

---

### SEC-H2. [HIGH] Open Redirect in Legacy Callback

**What:** In `auth_routes.py` (lines 139-145), a redirect uses string replacement on the full URL:

```python
return RedirectResponse(
    url=str(request.url).replace("/auth/callback", "/auth/callback/google"),
    status_code=307,
)
```

**How it works:** `str(request.url)` includes query parameters. An attacker could craft a URL where `/auth/callback` appears in a query parameter, causing the replacement to produce unexpected results or redirect to an attacker-controlled domain.

**Fix:** Use `request.url.path` for the replacement, or construct the target URL explicitly.

---

### SEC-H3. [HIGH] Dynamic SQL Column Names from User Input (Partial)

**What:** The PATCH endpoints in `repos.py`, `workspaces.py`, and `sessions.py` construct SQL with f-strings for column names. While repos and workspaces filter against `_UPDATABLE_COLUMNS` allowlists, `sessions.py` does NOT have this guard.

**How it works:**
```python
updates = body.model_dump(exclude_none=True)  # sessions.py uses exclude_none!
set_clause = ", ".join(f"{k} = ?" for k in updates)
db.execute(f"UPDATE sessions SET {set_clause} WHERE id = ?", values)
```

The `SessionUpdate` Pydantic model currently has only `model` as a field, so the risk is contained. But any field added to the model in the future would be directly injectable into the SQL without review.

**Fix:** Apply the same `_UPDATABLE_COLUMNS` allowlist pattern to `sessions.py`.

---

### SEC-H4. [HIGH] Empty Encryption Pepper Silently Disables Key Encryption

**What:** In `services/accounts.py` (line 180), when `ENCRYPTION_PEPPER` is empty, the DEK is stored as `NULL`. The application starts normally but API key encryption fails at runtime with a 500 error.

**Fix:** Add a startup validation that requires `ENCRYPTION_PEPPER` when running in production mode.

---

### SEC-M1. [MEDIUM] CORS Allows localhost in Production

**What:** In `main.py` (lines 44-50), `http://localhost:5173` is always in the allowed origins list, even in production, with `allow_credentials=True`.

**Fix:** Only include localhost origin when `settings.debug` is `True`.

---

### SEC-M2. [MEDIUM] Session Tokens Cannot Be Revoked (30-Day Validity)

**What:** Session tokens are signed JWTs with a 30-day max age and no server-side session store. A stolen token remains valid for the full 30 days. Logout only deletes the client-side cookie.

**Fix:** Add a `session_version` counter to users, embed it in tokens, and check it on each request. Increment on password change, account suspension, or explicit logout.

---

### SEC-M3. [MEDIUM] No Rate Limiting on Any Endpoint

**What:** There are no rate limiting mechanisms. The `/api/sessions/{id}/prompt` endpoint could drain all platform credit. Auth endpoints have no brute force protection.

**Fix:** Add `slowapi` rate limiting, especially on prompt submission and authentication endpoints.

---

### SEC-M4. [MEDIUM] Git Clone URL Allows SSH Scheme (Potential SSRF)

**What:** `services/git.py` allows `ssh://` and `git@` URL schemes, which could trigger connections to internal network hosts or arbitrary SSH servers.

**Fix:** For a SaaS platform, restrict to `https://` only. If SSH is needed, validate hostnames against a denylist.

---

### SEC-M5. [MEDIUM] `os.walk` May Follow Symlinks Outside Workspace

**What:** The file tree endpoint in `sessions.py` uses `os.walk()` on workspace paths. A malicious repository could contain symlinks pointing outside the workspace.

**Fix:** Add a check that each resolved path is within the workspace directory. Use `os.walk(followlinks=False)`.

---

### SEC-L1. [LOW] Sidecar Error Messages Leak Internal Details

**What:** Raw sidecar error messages (including internal exception text, file paths) are forwarded to clients via SSE.

**Fix:** Sanitize error messages before sending to clients.

---

### SEC-L2. [LOW] No Security Response Headers

**What:** No `Content-Security-Policy`, `X-Frame-Options`, `X-Content-Type-Options`, or `Strict-Transport-Security` headers are set.

**Fix:** Add a security headers middleware.

## Performance Findings

These findings come from the performance-oracle agent.

### PERF-C1. [CRITICAL] Synchronous SQLite Calls Block the Async Event Loop

**What:** The `prompt_session` endpoint in `stream.py` is declared as `async def`, but its `event_stream()` generator calls synchronous `sqlite3.connect()` and `.execute()` repeatedly:

```python
# Inside event_stream() -- an async generator running on the event loop:
with get_db_for_request(request) as db:
    db.execute("UPDATE messages SET content = ? ...")
    db.commit()
```

**How it works:** This happens every 10 streaming chunks (`_PERSIST_BATCH_SIZE`), on every result event, and in the `finally` block. Each call blocks the entire event loop for the duration of the SQLite I/O. All concurrent SSE streams and API handlers stall.

**Why this matters:** With just 5-10 concurrent streaming sessions, users will see visible latency spikes. The event loop is single-threaded -- one blocking call freezes everything.

**Fix:** Wrap all database operations in `event_stream()` with `await asyncio.to_thread(...)`.

---

### PERF-C2. [CRITICAL] Connection-Per-Call SQLite Pattern -- Excessive Churn

**What:** Every database operation opens a new connection and immediately closes it. A single prompt request opens and closes 7-8 separate connections:

1. AuthMiddleware: opens control DB to resolve tenant
2. `prompt_session`: opens user DB to look up session
3. `resolve_api_key_for_prompt` -> opens control DB
4. `get_user_dek` -> opens control DB again
5. Opens user DB to save user message
6. Repeatedly opens user DB during streaming (every 10 chunks)
7. Opens user DB in finally block

**Why this matters:** At 100 requests/second, this is 700-800 connection open/close cycles per second. SQLite connection setup includes PRAGMA execution, WAL initialization, and filesystem locks.

**Fix:** Implement connection pooling. At minimum, keep one connection open per request lifecycle rather than per operation.

---

### PERF-C3. [CRITICAL] Sidecar Sessions Never Evicted from Memory

**What:** The sidecar's `activeSessions` Map stores pi sessions indefinitely. Sessions are added during `warmupSession` and `processQuery` but never removed except during server shutdown.

**How it works:** Each session allocates an `AgentSession`, `AuthStorage`, `ModelRegistry`, `SessionManager`, `SettingsManager`, and `CodingTools` instance. The `socket.on("close")` handler only logs -- it does not clean up sessions.

**Why this matters:** Memory grows linearly with the number of unique sessions ever created. With 100 users running 5 sessions each, 500 session objects stay in memory permanently. The sidecar will eventually OOM.

**Fix:** Track socket-to-session ownership. Dispose sessions on socket close. Add TTL-based eviction for idle sessions (e.g., 30 minutes). Add a maximum session count cap.

---

### PERF-C4. [CRITICAL] Double Sidecar Connection Per Prompt Request

**What:** In `stream.py` (lines 145-151 and 193), every prompt request in tenant mode creates two separate Unix socket connections to the sidecar: one for `resolve_model()`, then a second for the actual query.

**Fix:** Resolve the model on the same connection used for the query.

---

### PERF-O1. DEK Unwrapping on Every API Key Lookup -- No Caching

**What:** Every call to `resolve_api_key_for_prompt` triggers HKDF key derivation (CPU-intensive) plus a control DB query to read the encrypted DEK.

**Fix:** Cache unwrapped DEKs per user_id with a 5-minute TTL using `cachetools.TTLCache`.

---

### PERF-O2. N+1 Query Pattern in Frontend Sidebar

**What:** Each `WorkspaceItem` component independently fetches its sessions via `GET /api/workspaces/{id}/sessions`. For 10 repos with 5 workspaces each, this fires 50 HTTP requests on page load.

**Fix:** Create a batch endpoint like `GET /api/repos/{id}/workspaces-with-sessions`.

---

### PERF-O3. Duplicate `/api/repos` Fetch on Page Load

**What:** Both `Sidebar.tsx` and `RepoList.tsx` independently fetch `/api/repos` on mount, causing the same API call twice.

**Fix:** Use a shared data context or SWR/TanStack Query with deduplication.

---

### PERF-O4. `os.walk` for File Tree is Unbounded

**What:** The `/api/sessions/{id}/tree` endpoint does a full synchronous `os.walk` with no depth limit, file count limit, or filtering of large directories like `node_modules`.

**Fix:** Use `git ls-files` instead. Add a file count cap (e.g., 5000). Filter `node_modules`, `.venv`, `__pycache__`.

---

### PERF-O5. AuthMiddleware Opens DB Connection on Every Request

**What:** Every authenticated request triggers a control DB query to resolve the tenant, before the handler even runs.

**Fix:** Cache user_id -> tenant mapping with a 60-second TTL.

---

### PERF-O6. Redundant SELECT After INSERT

**What:** All INSERT operations (repos, sessions, workspaces) do a `SELECT * WHERE rowid = ?` immediately after insert to return the created row.

**Fix:** Use SQLite's `INSERT ... RETURNING *` (available since SQLite 3.35.0) or construct the response from known values.

---

### PERF-O7. Legacy Migration Processes Data Row-by-Row

**What:** `_migrate_legacy_data` in `accounts.py` uses nested Python loops (repos -> workspaces -> sessions -> messages) instead of bulk SQL.

**Fix:** Use `ATTACH DATABASE` + `INSERT INTO ... SELECT FROM` for 10-100x faster migration.

---

### PERF-O8. Missing Index on `usage_log.session_id`

**What:** The `usage_log` table has an index on `user_id` but not `session_id`.

**Fix:** Add `CREATE INDEX IF NOT EXISTS idx_usage_session ON usage_log(session_id)`.

---

### CQ4. `HTTPException` Raised in Service Layer

**What:** `services/keys.py` raises `HTTPException` (a FastAPI-specific exception) directly from the service layer. Services should not know about HTTP status codes.

**Why this matters:** The service layer should raise domain-specific exceptions (e.g., `KeyNotFoundError`, `CreditExhaustedError`). The API layer should catch these and convert them to appropriate HTTP responses. This couples the service layer to FastAPI.

**Fix:** Define domain exceptions in `exceptions.py` and raise those from services. Catch them in API handlers.

---

### CQ5. `exclude_none` vs `exclude_unset` Bug in Sessions PATCH

**What:** In `sessions.py` (line 106), the PATCH handler uses `body.model_dump(exclude_none=True)` while repos and workspaces use `body.model_dump(exclude_unset=True)`.

**How it works:** `exclude_unset=True` means "skip fields the client did not include in the request body" -- correct for PATCH semantics. `exclude_none=True` means "skip fields with value None, even if the client explicitly sent `null`." This means a client cannot set a field to `null` in the sessions PATCH endpoint.

**Fix:** Change to `exclude_unset=True` for correct PATCH semantics.

---

### CQ6. 200-Line `prompt_session` Handler Mixes Multiple Concerns

**What:** The `prompt_session` function in `stream.py` is ~200 lines and handles: input validation, session lookup, model resolution, key resolution, SSE event formatting, database persistence, usage tracking, and error handling.

**Fix:** Extract helper functions for the distinct responsibilities: `_persist_chunk()`, `_finalize_stream()`, `_lookup_session()`, `_build_sse_event()`.

---

### CQ7. Module-Level Mutable State in `sidecar.py`

**What:** `SidecarClient` stores connection state (reader, writer, connected flag) as instance attributes but is created module-level via factory functions. If a connection drops and the caller creates a new client, the old writer/reader are not cleaned up.
