## Design Findings

### DESIGN-1: Keep the explicit-storage rule inside `RunnerStorageProfileSpec`

Source: `/skill:software-design`. Locations: `backend/src/yinshi/runner_agent.py:204` and `backend/src/yinshi/services/runners.py:287`.

The refactor removed helpers whose only job was to answer whether the selected storage profile must send explicit storage class values.

Before, the runner agent used this helper:

```python
def _requires_explicit_storage(profile: RunnerStorageProfileSpec) -> bool:
    """Return whether a profile requires explicit Archil storage class evidence."""
    return profile.value != _AWS_STORAGE_PROFILE

explicit_storage_required = _requires_explicit_storage(profile)
```

The control-plane service had the same rule under a different private name:

```python
def _profile_requires_explicit_storage(profile: RunnerStorageProfileSpec) -> bool:
    """Return whether a profile needs explicit Archil storage declarations."""
    return profile.value != _AWS_STORAGE_PROFILE

explicit_storage_required = _profile_requires_explicit_storage(profile)
```

Current code computes the rule inline in both modules:

```python
explicit_storage_required = profile.value != _AWS_STORAGE_PROFILE
```

Deleting those helpers is defensible because each helper had a larger interface than implementation. A reader gained little from jumping into a function that only wrapped one comparison.

The remaining design problem is ownership. `RunnerStorageProfileSpec` already stores profile-specific facts: default paths, allowed SQLite storage classes, allowed shared-files storage classes, and whether SQLite can sit under shared files. The rule about explicit storage classes belongs with those facts. When validation code says `profile.value != _AWS_STORAGE_PROFILE`, it knows too much about profile names.

A safer shape is to make the rule a field on the spec:

```python
@dataclass(frozen=True, slots=True)
class RunnerStorageProfileSpec:
    value: RunnerStorageProfile
    sqlite_storage: str
    shared_files_storage: str
    requires_explicit_storage: bool
    default_sqlite_dir: str
    default_shared_files_dir: str
    live_sqlite_on_shared_files: bool
    experimental: bool
    allow_sqlite_under_shared_files: bool
    allowed_sqlite_storage: frozenset[str]
    allowed_shared_files_storage: frozenset[str]
```

Then validation code reads the policy directly from the profile:

```python
explicit_storage_required = profile.requires_explicit_storage
```

That makes the profile object the single place to check when a new storage profile changes defaulting rules. Without that move, a future profile can accidentally pass runner-agent validation and fail control-plane validation, or the reverse, if one call site is updated and the other is missed.

### DESIGN-2: Keep a fallback class for unknown runner statuses

Source: `/skill:software-design`. Location: `frontend/src/components/CloudRunnerSection.tsx:108-109`.

The status badge moved from an `if` chain to a typed map.

Before:

```tsx
function runnerStatusClass(status: CloudRunnerStatus): string {
  if (status === "online") {
    return "border-green-500/40 bg-green-500/10 text-green-300";
  }
  if (status === "pending") {
    return "border-yellow-500/40 bg-yellow-500/10 text-yellow-200";
  }
  if (status === "revoked") {
    return "border-red-500/40 bg-red-500/10 text-red-300";
  }
  return "border-gray-600 bg-gray-800 text-gray-300";
}
```

After:

```tsx
const RUNNER_STATUS_CLASSES: Record<CloudRunnerStatus, string> = {
  offline: "border-gray-600 bg-gray-800 text-gray-300",
  online: "border-green-500/40 bg-green-500/10 text-green-300",
  pending: "border-yellow-500/40 bg-yellow-500/10 text-yellow-200",
  revoked: "border-red-500/40 bg-red-500/10 text-red-300",
};

function runnerStatusClass(status: CloudRunnerStatus): string {
  return RUNNER_STATUS_CLASSES[status];
}
```

The map improves compile-time coverage. If the local `CloudRunnerStatus` union gains a value, TypeScript can force a matching class entry. The weak spot is runtime API data. `api.get<CloudRunner | null>("/api/settings/runner")` gives TypeScript a type for the returned value, but it does not validate the JSON payload in the browser. If a newer backend sends `"degraded"`, JavaScript indexes the map with that string and receives `undefined`.

The old final `return` gave unknown values the offline-style badge. The map can keep that behavior:

```tsx
function runnerStatusClass(status: string): string {
  return RUNNER_STATUS_CLASSES[status as CloudRunnerStatus] ?? RUNNER_STATUS_CLASSES.offline;
}
```

A more explicit version validates the key first:

```tsx
function isCloudRunnerStatus(value: string): value is CloudRunnerStatus {
  return Object.prototype.hasOwnProperty.call(RUNNER_STATUS_CLASSES, value);
}

function runnerStatusClass(status: string): string {
  if (isCloudRunnerStatus(status)) {
    return RUNNER_STATUS_CLASSES[status];
  }
  return RUNNER_STATUS_CLASSES.offline;
}
```

That guard is useful because the API response is a boundary between separately deployed code. The UI should stay readable during partial rollouts or malformed responses.

const RUNNER_STATUS_CLASSES: Record<CloudRunnerStatus, string> = {

## Simplification Findings

### MINOR-1: `optionStatusClass` now only forwards to a map

Source: `/skill:code-simplicity-reviewer` in analysis-only mode. Location: `frontend/src/components/CloudRunnerSection.tsx:112-113`.

The option status helper used to contain the status-selection logic:

```tsx
function optionStatusClass(status: RunnerSetupOption["status"]): string {
  if (status === "Experimental") {
    return "border-yellow-500/40 bg-yellow-500/10 text-yellow-200";
  }
  return "border-green-500/40 bg-green-500/10 text-green-300";
}
```

After the refactor, the map contains the choice and the helper just indexes it:

```tsx
const OPTION_STATUS_CLASSES: Record<RunnerSetupOption["status"], string> = {
  Experimental: "border-yellow-500/40 bg-yellow-500/10 text-yellow-200",
  Supported: "border-green-500/40 bg-green-500/10 text-green-300",
};

function optionStatusClass(status: RunnerSetupOption["status"]): string {
  return OPTION_STATUS_CLASSES[status];
}
```

`RUNNER_OPTIONS` is local static data, so this status value comes from the same file rather than the network. A runtime fallback is less important here than it is for `runnerStatusClass`.

If the team wants fewer tiny helpers, the JSX can use the map directly:

```tsx
<span className={`rounded-full border px-2 py-0.5 text-[11px] ${OPTION_STATUS_CLASSES[option.status]}`}>
  {option.status}
</span>
```

Keeping the helper is also reasonable because it names the intent and mirrors `runnerStatusClass`. This is a cleanup preference, not a correctness issue.
