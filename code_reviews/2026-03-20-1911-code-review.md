# Code Review

## Summary

Reviewed the BYOP Pi config branch against `origin/main`, focusing on import/sync behavior, runtime isolation, and the new settings UI. I found two major behavioral regressions in the backend and one smaller frontend state-management issue.

## Design Findings

### MAJOR: Sync can fail permanently after the user disables any category

`sync_pi_config()` fetches and hard-resets the repo, but it never removes the existing `*.disabled` files and directories created by the category toggles. During re-application, `_rename_if_present()` raises if the disabled target already exists. That means the first sync after disabling `settings`, `prompts`, `instructions`, or any other category can fail with `Target path already exists` instead of completing successfully.

Relevant code:

- `backend/src/yinshi/services/pi_config.py:361`
- `backend/src/yinshi/services/pi_config.py:370`
- `backend/src/yinshi/services/pi_config.py:628`

### MAJOR: Imported settings still reach the sidecar when BYOP should be inactive

`resolve_agent_dir()` correctly disables Pi config usage when container isolation is off or the config is not `ready`, but `_resolve_execution_context()` still forwards `get_sidecar_settings_payload()` unconditionally. In practice, a previously imported `settings.json` can continue to affect host-side or non-ready sessions even though the feature is supposed to be inactive in those states.

Relevant code:

- `backend/src/yinshi/api/stream.py:201`
- `backend/src/yinshi/api/stream.py:228`
- `backend/src/yinshi/api/stream.py:255`
- `backend/src/yinshi/services/pi_config.py:719`

## Security Findings

### MAJOR: Deleted root instruction files remain active after sync

The mirrored `agent/AGENTS.md` and `agent/CLAUDE.md` files are copied into the repo as untracked files. On sync, `_mirror_instruction_files()` only copies files that still exist at the root; it never removes mirrored files when the root file was deleted upstream. Because `git reset --hard` does not remove untracked files, old instructions can keep applying indefinitely after the source repo has removed them.

Relevant code:

- `backend/src/yinshi/services/pi_config.py:319`
- `backend/src/yinshi/services/pi_config.py:350`
- `backend/src/yinshi/services/pi_config.py:636`

## Performance Findings

No material performance regressions stood out in this change set.

## Code Quality Findings

### MINOR: Category toggles can lose updates under quick successive clicks

`toggleCategory()` derives the next payload from the current `config.enabled_categories` snapshot and does not serialize requests or apply an optimistic state update first. Two fast checkbox clicks can therefore race and overwrite each other, leaving the server with a different final set than the user intended.

Relevant code:

- `frontend/src/hooks/usePiConfig.ts:161`

## Simplification Findings

The new backend module boundaries are generally reasonable. The biggest simplification opportunity is to centralize “is Pi config active for this request?” into one helper that gates both `agentDir` and imported settings together.

## Changes Applied

No code changes were applied as part of this review.
