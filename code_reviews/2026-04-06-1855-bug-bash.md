### Bug 1: Google OAuth: unverified email hijacks accounts

**Detail Bug ID**: `bug_14c45906-3669-4f7e-ba32-c0f7ec245c1a`
**File**: `backend/src/yinshi/api/auth_routes.py`
**Security vulnerability**: Yes
**Impact**: Critical | **Severity**: Critical | **Priority**: P0
**Status**: Fixed

#### What was the problem?

The Google OAuth callback accepted any email from a Google ID token without checking whether Google had verified that the user actually owns that email address. An attacker could create a Google account with a victim's email (without verifying it), log in via "Login with Google," and get linked to the victim's existing Yinshi account -- gaining full access to their workspaces, repos, and chat sessions.

#### What was changed?

**Before:**
```python
email = user_info["email"]

try:
    tenant = resolve_or_create_user(
        provider="google",
        provider_user_id=user_info["sub"],
        email=email,
        ...
    )
```

**After:**
```python
email = user_info.get("email")
if not email or not user_info.get("email_verified"):
    return RedirectResponse(url="/?error=email_not_verified")

try:
    tenant = resolve_or_create_user(
        provider="google",
        provider_user_id=user_info["sub"],
        email=email,
        ...
    )
```

#### How does the code work?

1. First, we safely extract the email using `.get()` instead of bracket access, which avoids a `KeyError` if the field is missing.
2. Then, we check two conditions: the email must be present, and `email_verified` must be truthy. Google includes `email_verified: true` or `email_verified: false` in ID tokens.
3. If either condition fails, the user is redirected to the login page with an `email_not_verified` error instead of being logged in.
4. Only after both checks pass does the code proceed to `resolve_or_create_user`, which links the Google identity to an existing or new account.

#### Why was this change needed?

Without this check, account hijacking was possible. The GitHub OAuth callback already filtered for verified emails, but the Google callback was missing the same safeguard. This fix makes both OAuth flows consistent in requiring email verification before account linking.

### Bug 2: Logout only clears cookie; server-side session remains valid

**Detail Bug ID**: `bug_a6706f29-ccfd-4fb7-a52c-434311515d18`
**File**: `backend/src/yinshi/api/auth_routes.py`
**Security vulnerability**: Yes
**Impact**: Critical | **Severity**: Critical | **Priority**: P0
**Status**: Fixed

#### What was the problem?

When a user clicked "Logout," the server only deleted the browser cookie. The auth session record in the database remained active (not revoked). If someone had captured the session token before logout -- for example, on a shared computer -- they could continue using it for up to 30 days.

#### What was changed?

**Before:**
```python
@router.post("/logout")
async def logout() -> RedirectResponse:
    """Clear session cookie."""
    response = RedirectResponse(url="/")
    _clear_session_cookie(response)
    return response
```

**After:**
```python
@router.post("/logout")
async def logout(request: Request) -> RedirectResponse:
    """Revoke the current auth session and clear the session cookie."""
    user_id = _current_user_id(request)
    if user_id is not None:
        revoke_auth_sessions(user_id)
    response = RedirectResponse(url="/")
    _clear_session_cookie(response)
    return response
```

#### How does the code work?

1. First, we extract the user ID from the session cookie using `_current_user_id(request)`, which decodes and verifies the token.
2. If the user is authenticated (user_id is not None), we call `revoke_auth_sessions(user_id)`, which updates the `auth_sessions` database table by setting `revoked_at = CURRENT_TIMESTAMP` on all active sessions for that user.
3. Finally, we clear the browser cookie as before and redirect to the home page.

#### Why was this change needed?

The codebase already had session revocation infrastructure (used by `/logout-all`) and checked `revoked_at` during token verification. The `/logout` endpoint simply was not updated to use it. Now both endpoints properly invalidate sessions server-side.

### Bug 3: Repo import: path traversal and repository hijacking

**Detail Bug ID**: `bug_e9c68d52-f371-4c0c-8ff8-711178fd17cd`
**File**: `backend/src/yinshi/api/repos.py`
**Security vulnerability**: Yes
**Impact**: Critical | **Severity**: Critical | **Priority**: P0
**Status**: Fixed

#### What was the problem?

The repo import endpoint used the user-provided repository `name` directly when constructing the clone directory path. A name like `../../etc/evil` would cause the clone to be placed outside the intended `repos/` directory. Additionally, two users importing repos with the same name could collide on the same directory.

#### What was changed?

**Before:**
```python
if tenant:
    clone_dir = str(Path(tenant.data_dir) / "repos" / body.name)
else:
    clone_dir = str(Path.home() / ".yinshi" / "repos" / body.name)
```

**After:**
```python
# Sanitize name to prevent path traversal.
safe_name = Path(body.name).name
if not safe_name or safe_name != body.name or ".." in body.name:
    raise HTTPException(
        status_code=400,
        detail="Invalid repository name (must be a simple directory name)",
    )

if tenant:
    clone_dir = str(Path(tenant.data_dir) / "repos" / safe_name)
else:
    clone_dir = str(Path.home() / ".yinshi" / "repos" / safe_name)
```

#### How does the code work?

1. First, `Path(body.name).name` extracts only the final component of a path -- stripping any directory traversal. For example, `../../evil` becomes `evil`, and `foo/bar` becomes `bar`.
2. We then verify three conditions: the result is not empty, it matches the original name exactly (catching any path separators), and the name does not contain `..`. If any check fails, the request is rejected with a 400 error.
3. Only the sanitized name is used when constructing the clone directory path.

#### Why was this change needed?

Without sanitization, a malicious user could clone repositories into arbitrary filesystem locations, or access another user's cloned repository by picking the same name. This is a foundational input validation issue at a system boundary.

### Bug 4: Git clone reuse without URL check

**Detail Bug ID**: `bug_0ce0f936-ddab-4993-a6da-b959fa3634e7`
**File**: `backend/src/yinshi/services/git.py`
**Security vulnerability**: Yes
**Impact**: High | **Severity**: Major | **Priority**: P0
**Status**: Fixed

#### What was the problem?

The `clone_repo` function reused any existing git repository at the destination directory without checking whether it pointed to the same remote URL. If two different repos mapped to the same destination, the second import would silently use the first repo's data.

#### What was changed?

**Before:**
```python
if dest_path.exists():
    if await validate_local_repo(dest):
        logger.info("Reusing existing clone at %s", dest)
        try:
            with _git_askpass_env(access_token) as env:
                await _run_git(["fetch", "--all"], cwd=dest, env=env)
        except GitError:
            pass
        return dest
```

**After:**
```python
if dest_path.exists():
    if await validate_local_repo(dest):
        # Verify the existing clone's remote matches the requested URL
        # before reusing it to prevent cross-repo data leakage.
        try:
            existing_remote = await _run_git(
                ["remote", "get-url", "origin"], cwd=dest,
            )
        except GitError:
            existing_remote = ""
        if existing_remote.strip().rstrip("/") != url.strip().rstrip("/"):
            raise GitError(
                f"Destination already contains a clone of a different repository"
            )
        logger.info("Reusing existing clone at %s", dest)
        try:
            with _git_askpass_env(access_token) as env:
                await _run_git(["fetch", "--all"], cwd=dest, env=env)
        except GitError:
            pass
        return dest
```

#### How does the code work?

1. First, we run `git remote get-url origin` in the existing directory to get the configured remote URL.
2. We normalize both URLs by stripping whitespace and trailing slashes, then compare them.
3. If the URLs do not match, we raise a `GitError` instead of silently reusing the wrong repository.
4. The function's docstring already claimed it checked for "matching remote" -- now the code actually does.

#### Why was this change needed?

The function's own docstring promised remote URL verification, but the implementation skipped it. This allowed cross-user data leakage: one user could read another user's private repository by importing a repo with the same name.

### Bug 5: Legacy migration leaks NULL-owner repos to every new user

**Detail Bug ID**: `bug_e2f93f63-8efc-4c68-aeee-e2713203477c`
**File**: `backend/src/yinshi/services/accounts.py`
**Security vulnerability**: Yes
**Impact**: High | **Severity**: Major | **Priority**: P0
**Status**: Fixed

#### What was the problem?

During the migration from single-tenant to multi-tenant mode, the migration query included `OR owner_email IS NULL`. This caused every repository without an explicit owner to be copied into every new user's database -- meaning private repos were leaked to unrelated users.

#### What was changed?

**Before:**
```python
repos = source.execute(
    "SELECT * FROM repos WHERE owner_email = ? OR owner_email IS NULL",
    (tenant.email,),
).fetchall()
```

**After:**
```python
repos = source.execute(
    "SELECT * FROM repos WHERE owner_email = ?",
    (tenant.email,),
).fetchall()
```

#### How does the code work?

1. The query now only selects repos where `owner_email` exactly matches the migrating user's email.
2. Repos with `NULL` owner_email are no longer migrated to any user.

#### Why was this change needed?

The `OR owner_email IS NULL` clause was inherited from legacy-mode listing (where all users shared a database and NULL-owner repos were visible to everyone). In the multi-tenant world, each user gets an independent database copy, so copying NULL-owner repos to every user creates unsynchronized duplicates and leaks data across tenants.

### Bug 7: Sidecar: JSON null crashes process (DoS)

**Detail Bug ID**: `bug_f9b866b0-1355-4c8b-9d9e-d23b0c78bb7f`
**File**: `sidecar/src/sidecar.js`
**Security vulnerability**: No
**Impact**: High | **Severity**: Major | **Priority**: P1
**Status**: Fixed

#### What was the problem?

Sending the literal JSON string `"null"` to the sidecar's Unix domain socket caused `JSON.parse` to return `null`. The `handleRequest` method then tried to destructure `null`, crashing the entire sidecar process with a `TypeError`. Any local process that could connect to the socket could kill the sidecar instantly.

#### What was changed?

**Before:**
```javascript
handleRequest(request, socket) {
  const { type, id } = request;
```

**After:**
```javascript
handleRequest(request, socket) {
  if (!request || typeof request !== "object") {
    sendToSocket(socket, { id: "unknown", type: "error", error: "Invalid request format" });
    return;
  }
  const { type, id } = request;
```

#### How does the code work?

1. Before destructuring, we check that `request` is truthy and is an object. This catches `null`, `undefined`, strings, numbers, booleans, and arrays.
2. If the check fails, we send an error response back to the socket instead of crashing, and return early.
3. Only valid object requests proceed to the destructuring and switch statement.

#### Why was this change needed?

A single malformed message could crash the sidecar for all active agent sessions. Input validation at the protocol boundary prevents denial-of-service from any local process that can connect to the socket.

### Bugs 8+9: CORS middleware order and OPTIONS handling

**Detail Bug IDs**: `bug_4ce8c56e-1790-44ca-a53a-0f7cf1011941`, `bug_8fcc9e42-d2c8-485a-9deb-31e580315320`
**Files**: `backend/src/yinshi/main.py`, `backend/src/yinshi/auth.py`
**Security vulnerability**: No
**Impact**: High | **Severity**: Major | **Priority**: P1
**Status**: Fixed

#### What was the problem?

In Starlette/FastAPI, middleware runs in reverse registration order -- the last `add_middleware` call becomes the outermost layer. `CORSMiddleware` was registered first (innermost) and `AuthMiddleware` last (outermost). This meant auth ran before CORS, blocking preflight `OPTIONS` requests with 401 and omitting CORS headers on error responses. The app was unusable from any cross-origin frontend when auth was enabled.

#### What was changed?

**main.py -- Before:**
```python
app.add_middleware(CORSMiddleware, ...)  # innermost (runs last)
app.add_middleware(SessionMiddleware, ...)
app.add_middleware(AuthMiddleware)       # outermost (runs first)
```

**main.py -- After:**
```python
# Middleware order: last registered = outermost = runs first.
app.add_middleware(SessionMiddleware, ...)
app.add_middleware(AuthMiddleware)
app.add_middleware(CORSMiddleware, ...)  # outermost (runs first)
```

**auth.py -- Added:**
```python
# Allow CORS preflight requests to pass through to CORSMiddleware.
if request.method == "OPTIONS":
    return await call_next(request)
```

#### How does the code work?

1. `CORSMiddleware` is now registered last, making it the outermost middleware. It intercepts `OPTIONS` requests first and adds CORS headers to all responses (including 401s from AuthMiddleware).
2. As defense-in-depth, `AuthMiddleware` also explicitly passes `OPTIONS` requests through without checking auth, since browsers never include credentials in preflight requests.

#### Why was this change needed?

Without this fix, any cross-origin deployment (e.g., frontend on `localhost:3000`, backend on `localhost:8000`) was completely broken when authentication was enabled. Every API request failed because the browser's preflight check was rejected.

### Bug 10: Streaming TOCTOU race on concurrent prompts

**Detail Bug ID**: `bug_02da5a67-65aa-4722-b223-847a50defba4`
**File**: `backend/src/yinshi/api/stream.py`
**Security vulnerability**: No
**Impact**: Medium | **Severity**: Major | **Priority**: P1
**Status**: Fixed

#### What was the problem?

The prompt endpoint checked `session["status"] == "running"` before calling `_resolve_execution_context` (which performs async I/O). During those yield points, a second concurrent request could pass the same check, and both would insert a user message and set the session to "running" -- causing duplicate messages and wasted resources.

#### What was changed?

**Before:**
```python
if session["status"] == "running":
    raise HTTPException(status_code=409, ...)

context = await _resolve_execution_context(...)

with get_db_for_request(request) as db:
    db.execute("INSERT INTO messages ...")
    db.execute("UPDATE sessions SET status = 'running' WHERE id = ?", ...)
    db.commit()
```

**After:**
```python
if session["status"] == "running":
    raise HTTPException(status_code=409, ...)

context = await _resolve_execution_context(...)

with get_db_for_request(request) as db:
    result = db.execute(
        "UPDATE sessions SET status = 'running' WHERE id = ? AND status = 'idle'",
        (session_id,),
    )
    if result.rowcount == 0:
        raise HTTPException(status_code=409, ...)

    db.execute("INSERT INTO messages ...")
    db.commit()
```

#### How does the code work?

1. The early check (`if session["status"] == "running"`) remains as a fast-path rejection.
2. After async work completes, we use an atomic SQL `UPDATE ... WHERE id = ? AND status = 'idle'`. SQLite serializes writes, so only one concurrent request can successfully update the row.
3. We check `result.rowcount`: if it's 0, another request already claimed the session, so we return 409 Conflict.
4. The message insert only happens after the atomic claim succeeds, preventing duplicate user messages.

#### Why was this change needed?

The original non-atomic read-then-write pattern allowed two requests to race through the guard. The atomic `UPDATE ... WHERE status = 'idle'` ensures exactly one winner, even under concurrent load.

### Bug 11: ENCRYPTION_PEPPER invalid hex crashes at runtime

**Detail Bug ID**: `bug_a3a4b73a-0a8e-4d44-b370-8cb555ca705f`
**File**: `backend/src/yinshi/config.py`
**Security vulnerability**: No
**Impact**: Medium | **Severity**: Minor | **Priority**: P2
**Status**: Fixed

#### What was the problem?

If `ENCRYPTION_PEPPER` was set to an invalid hex string (e.g., with a `0x` prefix or odd length), the app would start successfully but crash at runtime when the `encryption_pepper_bytes` property was first accessed -- during user login or registration.

#### What was changed?

**Before:**
```python
def _validate_settings(settings: Settings) -> None:
    if not _auth_is_enabled(settings):
        return
    if settings.secret_key:
        return
    raise RuntimeError("SECRET_KEY must be set when authentication is enabled")
```

**After:**
```python
def _validate_settings(settings: Settings) -> None:
    if _auth_is_enabled(settings) and not settings.secret_key:
        raise RuntimeError("SECRET_KEY must be set when authentication is enabled")

    if settings.encryption_pepper:
        try:
            bytes.fromhex(settings.encryption_pepper)
        except ValueError as exc:
            raise RuntimeError(
                f"ENCRYPTION_PEPPER must be a valid hex string: {exc}"
            ) from exc
```

#### How does the code work?

1. The `SECRET_KEY` validation logic was simplified to a single conditional.
2. A new block validates `ENCRYPTION_PEPPER` at startup by calling `bytes.fromhex()` in a `try/except`. If the hex is invalid, a clear `RuntimeError` is raised that names the config variable, making the misconfiguration immediately actionable.

#### Why was this change needed?

Fail-fast at startup is better than fail-at-runtime during user operations. The codebase already validated `SECRET_KEY` at startup but missed `ENCRYPTION_PEPPER`. The resulting runtime error (`non-hexadecimal number found`) did not mention the config variable name, making debugging harder.

### Bug 12: User settings race condition (UNIQUE constraint)

**Detail Bug ID**: `bug_cb5066c4-5805-4341-b8e0-045f69442073`
**File**: `backend/src/yinshi/services/user_settings.py`
**Security vulnerability**: No
**Impact**: Medium | **Severity**: Minor | **Priority**: P2
**Status**: Fixed

#### What was the problem?

Two functions (`_upsert_settings_row` and `set_pi_settings_enabled`) used a non-atomic SELECT-then-INSERT pattern. Under concurrent requests for the same new user, both could read "no row exists," both attempt INSERT, and the loser would crash with `sqlite3.IntegrityError: UNIQUE constraint failed`.

#### What was changed?

**_upsert_settings_row -- Before:**
```python
with get_control_db() as db:
    existing_row = db.execute("SELECT ...").fetchone()
    if existing_row is None:
        db.execute("INSERT INTO user_settings ...")
    else:
        db.execute("UPDATE user_settings SET ...")
    db.commit()
```

**_upsert_settings_row -- After:**
```python
with get_control_db() as db:
    db.execute(
        "INSERT INTO user_settings (user_id, pi_settings_json, pi_settings_enabled) "
        "VALUES (?, ?, ?) "
        "ON CONFLICT(user_id) DO UPDATE SET "
        "pi_settings_json = excluded.pi_settings_json, "
        "pi_settings_enabled = excluded.pi_settings_enabled",
        (normalized_user_id, serialized_payload, int(enabled)),
    )
    db.commit()
```

The same pattern was applied to `set_pi_settings_enabled`.

#### How does the code work?

1. `INSERT ... ON CONFLICT(user_id) DO UPDATE SET ...` is SQLite's atomic upsert. If the row already exists (conflict on the `user_id` unique constraint), it updates the existing row instead of failing.
2. `excluded.pi_settings_json` refers to the values that were proposed in the INSERT, making the update use the intended new values.
3. This single statement replaces the two-step SELECT-then-INSERT/UPDATE, eliminating the race window entirely.

#### Why was this change needed?

The codebase already used this pattern elsewhere (e.g., `auth_routes.py` for GitHub installations). Using it consistently prevents IntegrityError crashes on concurrent writes.

### Bug 13: Short/tool-only agent turns lost in conversation history

**Detail Bug ID**: `bug_9f3b0ed1-32b0-4428-8022-47a5391a3262`
**File**: `backend/src/yinshi/api/stream.py`
**Security vulnerability**: No
**Impact**: Medium | **Severity**: Minor | **Priority**: P2
**Status**: Fixed

#### What was the problem?

The streaming persistence logic only created an assistant message database row when the text chunk count reached the batch size (10). For short responses (fewer than 10 chunks), `assistant_msg_id` was never set, so the `result` event's `full_message` was never saved. For tool-only turns (no text at all), no assistant message row was created, causing the entire turn to vanish from history on page refresh.

#### What was changed?

**Before:**
```python
if data.get("type") == "result":
    usage_data = data.get("usage", {})
    result_provider = data.get("provider", result_provider)
    if assistant_msg_id:
        with get_db_for_request(request) as db:
            db.execute("UPDATE messages SET full_message = ? WHERE id = ?", ...)
            db.commit()
```

**After:**
```python
if data.get("type") == "result":
    usage_data = data.get("usage", {})
    result_provider = data.get("provider", result_provider)
    with get_db_for_request(request) as db:
        if assistant_msg_id is None:
            assistant_msg_id = uuid.uuid4().hex
            db.execute(
                "INSERT INTO messages (id, session_id, role, content, turn_id) "
                "VALUES (?, ?, 'assistant', ?, ?)",
                (assistant_msg_id, session_id, accumulated, turn_id),
            )
        db.execute("UPDATE messages SET full_message = ? WHERE id = ?", ...)
        db.commit()
```

#### How does the code work?

1. When the `result` event arrives (signaling the end of the agent's turn), we check whether an assistant message row already exists.
2. If not (`assistant_msg_id is None`), we create one now with whatever content has accumulated (which may be empty for tool-only turns). This ensures every agent turn has a corresponding database row.
3. We then unconditionally save the `full_message` (the complete result event), which was previously skipped for short responses.

#### Why was this change needed?

Users refreshing the page would see incomplete or entirely missing agent responses. Tool-only turns (where the agent used tools but produced no text) disappeared completely, making it impossible to track what the agent did.

### Bug 14: PATCH session model=null causes GET 500

**Detail Bug ID**: `bug_384e6377-0d61-4d34-a0f7-8f42106a8db8`
**File**: `backend/src/yinshi/models.py`
**Security vulnerability**: No
**Impact**: Medium | **Severity**: Minor | **Priority**: P2
**Status**: Fixed

#### What was the problem?

`SessionUpdate.model` accepted `None` (JSON `null`), which wrote `NULL` into the SQLite `model` column. But `SessionOut.model` was typed as a non-nullable `str`, so reading the session back caused a Pydantic `ValidationError` and HTTP 500. The session was effectively "bricked" -- inaccessible via the API.

```python
@field_validator("model")
@classmethod
def validate_model(cls, value: str | None) -> str | None:
```

#### Why was this change needed?

The API contract was inconsistent: the update model allowed null but the response model did not. Rejecting null at the input boundary prevents the database from entering a state that breaks reads.

### Bug 15: Pi config: category toggle during sync silently overwritten

**Detail Bug ID**: `bug_22a72c28-e90e-4e20-b747-f961a97b14bb`
**Files**: `frontend/src/hooks/usePiConfig.ts`, `backend/src/yinshi/services/pi_config.py`
**Security vulnerability**: No
**Impact**: Medium | **Severity**: Minor | **Priority**: P2
**Status**: Fixed

#### What was the problem?

The frontend allowed users to toggle categories while a sync was in progress. The backend had a TOCTOU race where the status was set to "syncing" non-atomically. A toggle could succeed through the race window, and then the sync would overwrite the change, silently losing the user's selection.

#### What was changed?

**Frontend -- Before:**
```typescript
if (updatingCategories) {
  return false;
}
```

### Bug 16: Lazy-loaded routes crash to white screen on chunk failure

**Detail Bug ID**: `bug_24d91235-a988-46c4-91e6-25c5c9c1332b`
**File**: `frontend/src/App.tsx` (new file: `frontend/src/components/ChunkErrorBoundary.tsx`)
**Security vulnerability**: No
**Impact**: Low | **Severity**: Minor | **Priority**: P3
**Status**: Fixed

#### What was the problem?

React.lazy() routes were wrapped in `<Suspense>` (which handles the loading state) but not in an `ErrorBoundary` (which handles errors). If a code-split chunk failed to download (e.g., network drop, deploy mismatch), the error propagated uncaught and the entire app rendered a white screen with no way to recover.

#### What was changed?

**New file -- ChunkErrorBoundary.tsx:**
```tsx
export default class ChunkErrorBoundary extends React.Component<
  React.PropsWithChildren,
  State
> {
  state: State = { hasError: false };

  static getDerivedStateFromError(): State {
    return { hasError: true };
  }

  handleRetry = () => {
    this.setState({ hasError: false });
  };

  render() {
    if (this.state.hasError) {
      return (
        <div style={{ padding: "2rem", textAlign: "center" }}>
          <p>Something went wrong loading this page.</p>
          <button onClick={this.handleRetry}>Try again</button>
        </div>
      );
    }
    return this.props.children;
  }
}
```

**App.tsx -- Before:**
```tsx
<Route path="/app" element={<Suspense><EmptyState /></Suspense>} />
```

**App.tsx -- After:**
```tsx
<Route path="/app" element={<ChunkErrorBoundary><Suspense><EmptyState /></Suspense></ChunkErrorBoundary>} />
```

#### How does the code work?

1. `ChunkErrorBoundary` is a React class component that uses the `getDerivedStateFromError` lifecycle method to catch any error thrown by its children.
2. When an error occurs (such as a `ChunkLoadError` from a failed lazy import), the component renders a friendly message with a "Try again" button.
3. Clicking "Try again" resets the error state, which causes React to re-render the children and re-attempt the lazy import.
4. Each lazy-loaded route is independently wrapped, so a failure on one route does not affect others.

#### Why was this change needed?

A white screen with no recovery option is a poor user experience. On unstable networks or during deployments (when chunk hashes change), users would lose the entire UI. The error boundary provides a clear message and a simple recovery path.
