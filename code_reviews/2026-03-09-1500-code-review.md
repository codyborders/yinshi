## Summary

This review covers the database connection management module and its tests, specifically the addition of the `_migrate()` function for schema migrations and `get_db_async()` for async database access.

**Key takeaways**:
- 1 Critical security finding (API key exposure in `.env`)

- 2 High-severity findings (missing auth on stream endpoints, fragile SQL column interpolation)
- Several medium/minor issues around connection management, test quality, and migration scalability

## Design Findings

### D-1: Connection Setup Duplication (Minor)

**What**: The logic to open a SQLite connection is written twice -- once in `get_db()` and once in `get_db_async()._connect()`.

**How it works**: Both functions do the same 3 steps:
1. Call `sqlite3.connect(path)`
2. Set `conn.row_factory = sqlite3.Row` (makes rows behave like dictionaries)
3. Run `PRAGMA foreign_keys = ON` (enables foreign key enforcement)

```python
# In get_db() -- lines 77-80:
settings = get_settings()
conn = sqlite3.connect(settings.db_path)
conn.row_factory = sqlite3.Row
conn.execute("PRAGMA foreign_keys = ON")

# In get_db_async()._connect() -- lines 91-94 (nearly identical):
settings = get_settings()
conn = sqlite3.connect(settings.db_path, check_same_thread=False)
conn.row_factory = sqlite3.Row
conn.execute("PRAGMA foreign_keys = ON")
```

**Why it matters**: If someone needs to change connection setup (e.g., adding `PRAGMA busy_timeout`), they must remember to change it in both places. This is called "information leakage" in software design -- the same decision (how to configure a connection) lives in two locations.

**Recommendation**: Extract a shared `_open_connection(db_path, check_same_thread=True)` helper.

---

### D-2: Misleading PRAGMAs in SCHEMA_SQL (Minor)

**What**: `SCHEMA_SQL` contains both `PRAGMA journal_mode = WAL` and `PRAGMA foreign_keys = ON` (lines 14-15), but they behave very differently:

- `journal_mode = WAL` is **persistent** -- set once, it sticks forever for that database file
- `foreign_keys = ON` is **per-connection** -- must be set every time you open a connection

**How it works**: `SCHEMA_SQL` is run via `conn.executescript()`, which auto-commits before executing. The `foreign_keys` pragma set inside `executescript` does NOT carry over to later operations on the same connection. That's why `get_db()` sets it again on line 80.

**Why it matters**: Having `PRAGMA foreign_keys = ON` in `SCHEMA_SQL` suggests to a reader that setting it once during schema creation is sufficient. It isn't. The one in `get_db()` is the one that actually matters.

**Recommendation**: Remove `PRAGMA foreign_keys = ON` from `SCHEMA_SQL`. Add a comment in `get_db()` explaining why the per-connection pragma is necessary.

### D-3: Migration Pattern Will Not Scale (Medium)

**What**: The `_migrate()` function checks for a specific column by scanning `PRAGMA table_info`:

```python
def _migrate(conn: sqlite3.Connection) -> None:
    columns = [row[1] for row in conn.execute("PRAGMA table_info(repos)").fetchall()]
    if "owner_email" not in columns:
        logger.info("Migrating: adding owner_email column to repos")
        conn.execute("ALTER TABLE repos ADD COLUMN owner_email TEXT")
```

**How it works**: Every time the app starts, it fetches all column names from the `repos` table, checks if `owner_email` is present, and adds it if missing. This works perfectly for one migration.

**Why it matters**: When you need a second migration (say, adding an `is_archived` column to workspaces), you'd add another `PRAGMA table_info(workspaces)` check and another `ALTER TABLE`. After 5-10 migrations, this function becomes an unordered pile of ad-hoc checks with no way to know which migrations ran or in what order.

**Recommendation**: Use a version-based migration approach:

```python
_MIGRATIONS: list[tuple[int, str]] = [
    (1, "ALTER TABLE repos ADD COLUMN owner_email TEXT"),
]

def _migrate(conn: sqlite3.Connection) -> None:
    conn.execute("CREATE TABLE IF NOT EXISTS schema_version (version INTEGER NOT NULL)")
    row = conn.execute("SELECT MAX(version) FROM schema_version").fetchone()
    current = row[0] or 0
    for version, sql in _MIGRATIONS:
        if version > current:
            logger.info("Applying migration %d", version)
            conn.execute(sql)
            conn.execute("INSERT INTO schema_version (version) VALUES (?)", (version,))
    conn.commit()
```

This tracks what has already run, executes migrations in order, and makes adding new migrations a one-line change.

## Security Findings

### S-1: Live API Key in `.env` (CRITICAL)

**What**: The `.env` file contains a plaintext MiniMax API key:

```
MINIMAX_API_KEY=sk-cp-lp5GXzTL...
```

**How it works**: The `.env` file is loaded by `pydantic_settings.BaseSettings` at startup. While `.env` is in `.gitignore` (good), the key sits in plaintext on disk.

**Why it matters**: Anyone with filesystem access (shared machine, backup, compromised account) can steal this key and make API calls at your expense. The key format (`sk-cp-...`) indicates it is a real, active credential.

**Recommendation**:
1. **Rotate this key immediately** -- generate a new one from MiniMax and revoke the old one
2. Audit git history (`git log -p -- .env`) to confirm it was never committed
3. Consider using a secrets manager for production credentials

---

### S-2: Missing Auth on Stream Endpoints (MEDIUM)

**What**: The `prompt_session` and `cancel_session` endpoints in `stream.py` do not check resource ownership.

**How it works**: These endpoints take a `session_id` from the URL path but never call `get_user_email(request)` or `check_owner()`. If auth is enabled, any authenticated user who can guess a session ID (128-bit hex, so hard but not impossible) can send prompts to or cancel another user's agent session.

**Why it matters**: This is a horizontal privilege escalation. The other endpoints (`sessions.py`, `workspaces.py`, `repos.py`) all have ownership checks. The stream endpoints were missed.

**Recommendation**: Add the same ownership check pattern used in `sessions.py`:

```python
email = get_user_email(request)
# After fetching session, join through to get owner_email:
# SELECT s.*, r.owner_email FROM sessions s
#   JOIN workspaces w ON s.workspace_id = w.id
#   JOIN repos r ON w.repo_id = r.id WHERE s.id = ?
check_owner(row["owner_email"], email)
```

### S-3: Dynamic SQL Column Names in `repos.py` (HIGH)

**What**: The `update_repo` endpoint builds SQL dynamically by interpolating column names:

```python
# repos.py lines 108-110:
set_clause = ", ".join(f"{k} = ?" for k in updates)
values = list(updates.values()) + [repo_id]
db.execute(f"UPDATE repos SET {set_clause} WHERE id = ?", values)
```

**How it works**: The values are safely parameterized with `?` placeholders (good). But the column names (`k`) are inserted directly into the SQL string. Today, this is safe because `updates` is filtered through `_UPDATABLE_COLUMNS = {"name", "custom_prompt"}` (line 19), so only those two known-safe names can appear.

**Why it matters**: If a future developer adds a field to `_UPDATABLE_COLUMNS` without thinking about SQL safety, or removes the allowlist during refactoring, this becomes a SQL injection vector. The code is correct today but fragile.

**Recommendation**: Add an explicit regex check before interpolation:

```python
import re
for k in updates:
    if not re.match(r'^[a-z_]+$', k):
        raise ValueError(f"Invalid column name: {k}")
```

### S-4: Weak `check_owner` Logic (MEDIUM)

**What**: The ownership check silently allows access when either email is `None`:

```python
def check_owner(owner_email: str | None, user_email: str | None) -> None:
    if user_email and owner_email and owner_email != user_email:
        raise HTTPException(status_code=403, detail="Not authorized")
```

**How it works**: If `owner_email` is `None` (pre-migration repos), access is allowed. If `user_email` is `None` (auth disabled), access is allowed. Both are intentional, but the logic is implicit.

**Why it matters**: If auth middleware has a bug or a new open endpoint is added, the silent `None` bypass could grant unintended access. The function should be explicit about when bypasses are acceptable and log when they happen.

---

### S-5: No Input Length Limits on Most Models (LOW)

**What**: `PromptRequest.prompt` correctly has `Field(max_length=100_000)`, but `RepoCreate.name`, `RepoCreate.remote_url`, `SessionCreate.model`, and other string fields have no length constraints.

**Why it matters**: An attacker could submit extremely long strings to bloat the database or cause memory issues.

**Recommendation**: Add `Field(max_length=...)` to all string fields in Pydantic models.

---

### S-6: Git Errors Leak Internal Paths (LOW)

**What**: Git stderr output is propagated to HTTP responses:

```python
# git.py line 60:
raise GitError(f"git {args[0]} failed: {stderr.decode().strip()}")

# repos.py line 63:
raise HTTPException(status_code=400, detail=str(e))
```

**Why it matters**: Error messages may contain filesystem paths like `/Users/user/.yinshi/repos/...`, revealing internal directory structure.

**Recommendation**: Log the full error server-side; return a generic message to the client.

## Performance Findings

### P-1: `get_db_async()` Blocks the Event Loop (CRITICAL)

**What**: `get_db_async()` wraps `sqlite3.connect()` and `conn.close()` in `asyncio.to_thread()`, but yields a raw synchronous `sqlite3.Connection`.

```python
@asynccontextmanager
async def get_db_async() -> AsyncIterator[sqlite3.Connection]:
    def _connect() -> sqlite3.Connection:
        settings = get_settings()
        conn = sqlite3.connect(settings.db_path, check_same_thread=False)
        conn.row_factory = sqlite3.Row
        conn.execute("PRAGMA foreign_keys = ON")
        return conn

    conn = await asyncio.to_thread(_connect)  # Only THIS is async
    try:
        yield conn  # Caller gets a SYNC connection
    finally:
        await asyncio.to_thread(conn.close)  # Only THIS is async
```

**How it works**: Only the `connect()` and `close()` calls run in a background thread. Every `.execute()`, `.fetchall()`, and `.commit()` call the caller makes on the yielded connection runs **synchronously on the event loop thread**, blocking all other async operations.

**Why it matters**: This is architecturally misleading. A developer using `async with get_db_async() as conn:` reasonably expects all operations on `conn` to be non-blocking. Instead, they silently block the event loop. In a streaming scenario with 10+ concurrent users, one slow query blocks SSE delivery to everyone.

**Recommendation**: Either:
- **Option A** (preferred): Use `aiosqlite`, which wraps every operation in a thread automatically
- **Option B**: Remove `get_db_async()` entirely and convert async handlers to plain `def` so FastAPI runs them in its threadpool (where blocking is acceptable)

---

### P-2: Connection Churn -- New Connection Per Operation (MAJOR)

**What**: Every `get_db()` call creates a brand new `sqlite3.connect()` and closes it when the context manager exits.

**How it works**: In `stream.py`, a single streaming session opens 5+ separate connections: initial lookup, save user message, ~10 batch persists (one per 10 chunks), result finalize, and a final flush in the `finally` block.

**Why it matters**: Connection setup is ~0.5-1ms per open/close. For a 100-chunk streaming response, that's 12+ connection cycles = ~12ms of pure overhead. More importantly, under 10+ concurrent streaming sessions, the write lock contention causes `SQLITE_BUSY` errors because there's no retry/timeout strategy.

**Recommendation**:
1. **Immediate**: Add `PRAGMA busy_timeout = 5000` to `get_db()` (one-line fix)
2. **Short-term**: Reuse a single connection within `event_stream()` for all persistence operations
3. **Long-term**: Consider a module-level connection singleton (appropriate for SQLite)

---

### P-3: Sync `get_db()` Used Inside `async def` Handlers (MAJOR)

**What**: Several `async def` handlers call `get_db()` directly:
- `repos.py`: `async def import_repo` (line 46)
- `workspaces.py`: `async def create_workspace` (line 37), `async def remove_workspace` (line 54)

**How it works**: When an `async def` handler calls `with get_db() as db:`, the `sqlite3.connect()`, `.execute()`, and `.commit()` calls all run on the event loop thread. This blocks ALL other coroutines -- no SSE chunks are sent, no health checks respond, no other requests are processed.

Note: Some handlers like `list_sessions`, `get_session` in `sessions.py` are correctly defined as plain `def`, which causes FastAPI to run them in a threadpool where blocking is safe.

**Why it matters**: With 10+ concurrent users, a single blocking `get_db()` call stalls the entire server.

**Recommendation**: Either convert these to plain `def` (so FastAPI threads them) or use `asyncio.to_thread()` around the entire DB-using block.

---

### P-4: No `busy_timeout` Pragma (MEDIUM)

**What**: Neither `get_db()` nor `get_db_async()` set `PRAGMA busy_timeout`.

**How it works**: Without `busy_timeout`, when one connection holds the write lock and another tries to write, SQLite immediately raises `sqlite3.OperationalError: database is locked` instead of waiting and retrying.

**Why it matters**: With concurrent streaming sessions doing batch persists, this will cause sporadic "database is locked" errors that are difficult to reproduce and debug.

**Recommendation**: Add one line to `get_db()`:

```python
conn.execute("PRAGMA busy_timeout = 5000")
```

This tells SQLite to wait up to 5 seconds for the write lock before raising an error.

## Code Quality Findings

### Q-1: `test_init_db_uses_context_manager` Tests Nothing Meaningful (MEDIUM)

**What**: This test (lines 95-100) claims to verify that `init_db` doesn't leak connections:

```python
def test_init_db_uses_context_manager(db):
    """init_db should not leak connections (uses get_db context manager)."""
    tables = db.execute(
        "SELECT name FROM sqlite_master WHERE type='table'"
    ).fetchall()
    assert len(tables) > 0
```

**How it works**: It just checks that tables exist -- the same thing `test_init_db_creates_tables` already tests. It doesn't actually verify connection lifecycle or detect leaks.

**Why it matters**: Misleading tests create false confidence. The docstring says "should not leak connections" but the test body can't detect a leak.

**Recommendation**: Either delete this test or rewrite it to actually test connection lifecycle.

### Q-3: Missing `cache_clear` on Failure Paths (MEDIUM)

**What**: Both `test_init_db_migrates_owner_email_column` and `test_get_db_async` call `get_settings.cache_clear()` at the end, but if the test raises an exception before that line, the cache is left polluted.

**How it works**: `get_settings()` is `@lru_cache`-decorated, so once called with a monkeypatched `DB_PATH`, the cached value persists until explicitly cleared. If a test fails mid-execution, subsequent tests see a stale `DB_PATH`.

**Why it matters**: Can cause cascading test failures that are hard to debug -- the second test fails with a confusing "file not found" error because it's using the first test's temp directory.

**Recommendation**: Use `try/finally` to ensure cleanup:

```python
try:
    # ... test body ...
finally:
    get_settings.cache_clear()
```

Or better, create a fixture that handles this automatically.

---

### Q-4: No Error Handling in `init_db` (MEDIUM)

**What**: `init_db()` calls `executescript` and `_migrate()` but catches no exceptions.

```python
def init_db() -> None:
    settings = get_settings()
    logger.info("Initializing database at %s", settings.db_path)
    with get_db() as conn:
        conn.executescript(SCHEMA_SQL)
        _migrate(conn)
    logger.info("Database initialized")
```

**Why it matters**: If migration fails, the error propagates as a raw `sqlite3.Error` with no context about what was being attempted.

**Recommendation**: Wrap in try/except to add context:

```python
try:
    with get_db() as conn:
        conn.executescript(SCHEMA_SQL)
        _migrate(conn)
except sqlite3.Error as exc:
    logger.error("Database initialization failed: %s", exc)
    raise
```
