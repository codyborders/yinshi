The change introduces a `ContainerManager` that gives each user a dedicated Docker container for their sidecar process, with dropped capabilities, resource limits, and a restricted bridge network. **Key takeaways**:

1. The `ContainerManager` is architecturally sound -- deep module with a simple interface hiding Docker complexity.
2. The most urgent issues are **synchronous Docker SDK calls blocking the async event loop** (affects all concurrent users) and **missing socket directory permissions** (local privilege escalation).

4. The streaming endpoint (`prompt_session`) has grown into a 250-line god-function that needs decomposition.

## Design Findings

These findings evaluate the code against principles from "A Philosophy of Software Design" -- focusing on abstraction depth, information hiding, interface design, and complexity red flags.

### D1 -- ContainerManager is a well-designed deep module (POSITIVE)

**What**: `ContainerManager` hides significant complexity (Docker API calls, per-user locking, socket polling, idle reaping, container lifecycle) behind a 3-method public interface: `ensure_container()`, `touch()`, `destroy_container()`.

**How it works**: Callers pass a `user_id` and `data_dir`, and get back a `ContainerInfo` with just a `socket_path`. The caller never touches the Docker SDK, never manages locks, and never worries about stopped containers -- `ensure_container` transparently replaces them.

**Why this is good**: This follows the "pull complexity downward" principle. The interface is simpler than the implementation, which is the hallmark of a deep module. The `ContainerInfo` dataclass exposes only what callers need.

### D2 -- `prompt_session` is a conjoined god-function (MAJOR)

**What**: The `prompt_session` endpoint in `stream.py` (lines 162-395) is approximately 250 lines including the nested `event_stream` generator. It handles session lookup, authorization, workspace path validation, container orchestration, model resolution, API key resolution, user message persistence, SSE streaming with batched DB writes, usage recording, and session cleanup.

**How it works**: The function has two phases: (1) a setup phase before the `return StreamingResponse(...)` that validates, resolves keys, and persists the user message; (2) an inner `event_stream()` async generator that creates the sidecar connection, streams events, persists assistant messages in batches, and cleans up in a `finally` block. These two phases share state via closure variables (`workspace_path`, `model`, `api_key`, `sidecar_socket`, `effective_cwd`, `tenant`, etc.).

**Why this is a problem**: This function exhibits multiple red flags:
- **Conjoined methods**: Understanding `event_stream` requires understanding `prompt_session`'s setup. They share 10+ closure variables.
- **Different layer, same abstraction**: The function mixes HTTP routing concerns (request parsing, status codes), infrastructure concerns (container orchestration), business logic (key resolution, credit checking), and I/O concerns (DB persistence, SSE formatting).
- **Hard to test**: The integration tests require deep mocking of sidecar connections, Docker, and the entire request context. Individual concerns cannot be tested in isolation.

**Recommendation**: Extract into separate functions/services:
1. A `resolve_execution_context()` that returns `(socket_path, effective_cwd, api_key, key_source, provider)` -- handles container orchestration + key resolution
2. A `persist_turn()` service that handles batched message persistence
3. The SSE generator itself, which only handles sidecar communication and event formatting

### D3 -- Duplicated path-containment logic across 3 locations (MEDIUM)

**What**: The pattern `os.path.realpath(path).startswith(os.path.realpath(base) + os.sep)` appears independently in:
- `stream.py:_path_inside()` (lines 96-100)
- `stream.py:_remap_path()` (lines 126-128)
- `repos.py:_validate_local_path()` (lines 34-36)

**How it works**: Each function resolves symlinks via `realpath()` and checks if the resolved path starts with the base directory path plus a separator. The check also handles the edge case where `path == base`.

**Why this is a problem**: This is the "repetition" red flag. If one location is updated (e.g., to handle additional edge cases like null bytes in paths), the others must be updated in lockstep. The three implementations also use slightly different APIs (`os.path` vs `pathlib.Path`), which could lead to subtle divergence.

**Recommendation**: Extract to a shared utility function in a `utils/paths.py` module:
```python
def is_path_inside(path: str, base: str) -> bool:
    """Check if path is inside base after resolving symlinks."""
    resolved = os.path.realpath(path)
    resolved_base = os.path.realpath(base)
    return resolved == resolved_base or resolved.startswith(resolved_base + os.sep)
```

### D4 -- `_remap_path` could call `_path_inside` instead of duplicating logic (MINOR)

**What**: `_remap_path` (stream.py:122-133) re-implements the same "is path inside base" check that `_path_inside` already provides, then does the remapping. The containment check in `_remap_path` could delegate to `_path_inside`.

**Why**: Reduces duplication and makes the relationship between validation and remapping explicit.

### D5 -- Sidecar connection created and discarded just for `resolve_model` (MEDIUM)

**What**: In `prompt_session` (stream.py:209-214), a sidecar connection is created, used solely to call `resolve_model()`, then immediately disconnected. A second connection is created later in `event_stream()` for the actual query.

**How it works**: The first connection sends a `resolve` message to the sidecar to determine the provider (e.g., "minimax" or "anthropic") for a given model key. This provider information is needed to resolve the API key before streaming begins. The second connection then sends `warmup` and `query` messages.

**Why this is a problem**: Two full connection handshakes (connect + init_status read + disconnect) per prompt request. The resolve could be done on the same connection used for the query, or the model-to-provider mapping could be maintained on the backend side to avoid the extra round-trip entirely.

### D6 -- `SidecarClient` does not implement async context manager (MINOR)

**What**: `SidecarClient` manages a socket connection and requires explicit `disconnect()` calls. Every usage site has a `try/finally` with `disconnect()`.

**Why**: Implementing `__aenter__`/`__aexit__` would make the code safer and more Pythonic, following the project's own PYTHON.md guidance on context managers.

## Security Findings

These findings come from the security-sentinel agent's audit of the container isolation implementation.

### S1 -- CRITICAL: Path traversal in socket directory creation

**File**: `container.py:100-101`

**What**: The `user_id` parameter is interpolated directly into a filesystem path without validation:

```python
# BEFORE (current code)
socket_dir = os.path.join(s.container_socket_base, user_id)
os.makedirs(socket_dir, exist_ok=True)
```

**How it works**: `os.path.join("/var/run/yinshi", user_id)` constructs the socket directory path. If `user_id` contained `../../etc`, the resulting path would be `/etc` rather than a subdirectory of the socket base.

**Why this matters**: Although `user_id` is currently generated as `secrets.token_hex(16)` (32 hex chars), the `ensure_container()` method accepts any string. A bug in the account system, a test, or a future code path could produce a malicious `user_id`. The same un-validated `user_id` is also used in the Docker volume mount (line 114), container name (line 108), and Docker label (line 128).

**Recommendation**: Validate `user_id` at the boundary:
```python
import re
if not re.fullmatch(r"[0-9a-f]{32}", user_id):
    raise ValueError(f"Invalid user_id format: {user_id!r}")
```

### S2 -- HIGH: Missing socket directory permissions

**File**: `container.py:101`

**What**: `os.makedirs(socket_dir, exist_ok=True)` creates the directory with the default umask (typically `0o022`), making it world-readable.

**How it works**: On a shared host, any local process can connect to the Unix socket at `/var/run/yinshi/{user_id}/sidecar.sock`. The sidecar accepts any connection on the socket without authentication. An attacker who connects can send `query` messages with prompts and receive responses, or send `warmup` messages to pre-create sessions.

**Why this matters**: The API key is sent in plaintext over this socket as part of the `options.apiKey` field. A local attacker who connects to another user's socket gets the victim's API key.

**Recommendation**: Set explicit permissions:
```python
os.makedirs(socket_dir, mode=0o700, exist_ok=True)
```

### S3 -- HIGH: Entire data_dir mounted read-write into containers

**File**: `container.py:115-118`

**What**: The user's entire data directory (containing their SQLite database, all repository clones, and all workspace worktrees) is bind-mounted into the container at `/data` with read-write access.

**How it works**: The pi coding agent running inside the container has shell access through its tools (`createCodingTools(cwd)`). It can read/write any file under `/data`, including `yinshi.db` (the user's session/message database).

**Why this matters**: A prompt injection attack or a malicious repository could instruct the agent to read the SQLite database and exfiltrate session history, API key references, or other metadata. The agent could also corrupt or delete the database.

**Recommendation**: Mount only the specific workspace directory (not the entire `data_dir`) into the container. The backend manages DB access directly, so the database should not be mounted at all.

### S4 -- HIGH: Docker network without inter-container isolation

**File**: `container.py:52`

**What**: All sidecar containers are placed on the same `yinshi-sidecar-net` bridge network, created without the `internal=True` flag.

**How it works**: Docker bridge networks allow containers on the same network to communicate freely with each other. Without `internal=True`, containers can also reach the internet. A compromised sidecar in one user's container can scan and attack other users' sidecars over the shared network.

**Why this matters**: Cross-tenant lateral movement. The plan document specifies restricted outbound traffic, but the code does not enforce it.

**Recommendation**: Create the network with isolation:
```python
self._docker.networks.create(_SIDECAR_NET, driver="bridge", internal=True)
```
For full isolation, consider per-user networks.

### S5 -- HIGH: Information disclosure via exception propagation

**File**: `stream.py:206-207`

**What**: When container start fails, the raw Docker API error is returned to the client:

```python
except (ContainerStartError, ContainerNotReadyError) as exc:
    raise HTTPException(status_code=503, detail=str(exc))
```

**How it works**: `ContainerStartError` wraps the Docker API error (container.py:132): `f"Failed to start container for user {user_id[:8]}: {exc}"`. Docker errors can contain host paths, Docker daemon version, resource limits, and network names.

**Why this matters**: Information disclosure of internal infrastructure details to any authenticated user.

**Recommendation**: Return a generic error message:
```python
raise HTTPException(status_code=503, detail="Agent environment temporarily unavailable")
```

### S6 -- MEDIUM: Container isolation bypass for allowed_repo_base paths

**File**: `stream.py:199`

**What**: Container isolation only activates when the workspace path is inside the tenant's `data_dir`:
```python
if container_mgr and _path_inside(workspace_path, tenant.data_dir):
```
Paths under `allowed_repo_base` (local imports) skip containerization entirely.

**How it works**: `_validate_workspace_path` allows paths in either `data_dir` or `allowed_repo_base`. But the container guard only checks `data_dir`. So local-import workspaces use the shared, non-containerized sidecar with full host filesystem access.

**Why this matters**: In multi-tenant mode, this means local imports completely bypass the container isolation that this entire change is designed to provide.

**Recommendation**: Either disable local imports when container isolation is enabled, or extend the container logic to handle `allowed_repo_base` paths.

### S7 -- MEDIUM: Orphaned containers on process crash

**File**: `container.py:40`

**What**: Container tracking is in-memory only (`self._containers` dict). If the FastAPI process crashes, all tracking is lost, and running containers are orphaned.

**How it works**: The `destroy_all()` shutdown hook only cleans up containers in the dict at shutdown time. An OOM kill or segfault skips the hook entirely.

**Recommendation**: On startup, query Docker for containers with the `yinshi.user_id` label and destroy them:
```python
stale = self._docker.containers.list(filters={"label": "yinshi.user_id"})
for c in stale:
    c.remove(force=True)
```

### S8 -- MEDIUM: Container name collision

**File**: `container.py:108`

**What**: Container names use only the first 12 characters of `user_id`: `f"yinshi-sidecar-{user_id[:12]}"`. Docker requires unique container names. Two users whose IDs share the first 12 hex chars would collide.

**Recommendation**: Use the full `user_id` in the container name.

### S9 -- MEDIUM: TOCTOU race on socket existence check

**File**: `container.py:150-161`

**What**: Between when `os.path.exists(socket_path)` returns `True` and when `SidecarClient.connect()` opens the socket, a local attacker could replace the socket with a symlink. This is mitigated by fixing S2 (restrict directory permissions to `0o700`).

### S10 -- LOW: Default sidecar socket in /tmp

**File**: `config.py:49`

**What**: Non-containerized mode defaults to `/tmp/yinshi-sidecar.sock`, which is world-writable. A local attacker could pre-create a symlink at this path.

**Recommendation**: Default to a path under a restricted directory (e.g., `/var/run/yinshi/sidecar.sock`).

### S11 -- LOW: No rate limiting on container creation

**File**: `container.py:62-76`

**What**: No cooldown between container creation cycles per user. A malicious user could rapidly trigger start/stop cycles, exhausting Docker daemon resources.

## Performance Findings

These findings come from the performance-oracle agent's analysis of bottlenecks, blocking calls, and scalability concerns.

### P1 -- HIGH: Synchronous Docker API calls block the event loop

**File**: `container.py:78-93`

**What**: `_is_running()` and `_remove_container()` call the Docker SDK synchronously (HTTP calls to the Docker daemon) within async functions.

**How it works**: `self._docker.containers.get(container_id)` and `c.remove(force=True)` are blocking HTTP calls to the Docker daemon API. They typically take 50-500ms each. These are called from `ensure_container()` (which checks if an existing container is running) and from `reap_idle()` (which removes idle containers in a loop).

**Why this matters**: During a reaper cycle with N idle containers, the event loop is blocked for N * 50-500ms. With 10 idle containers, that is 0.5-5 seconds where no SSE events are delivered to any connected client and no new HTTP requests are handled.

**Scalability impact**:

| Users | Containers to reap | Event loop blocked |
|-------|-------------------|--------------------|
| 10    | ~2-3              | negligible         |
| 100   | ~10-20            | 0.5-10s            |
| 1000  | ~100+             | 5-50s              |

**Recommendation**: Wrap all synchronous Docker calls in `asyncio.to_thread()`, matching the pattern already used for `containers.run` in `_create_container`:
```python
async def _is_running(self, container_id: str) -> bool:
    try:
        c = await asyncio.to_thread(self._docker.containers.get, container_id)
        return c.status == "running"
    except docker.errors.NotFound:
        return False
```

### P2 -- HIGH: Redundant sidecar connection for model resolution

**File**: `stream.py:209-214`

**What**: Every prompt request in tenant mode creates a throwaway sidecar connection just for `resolve_model()`, then disconnects. A second connection is created for the actual query.

**How it works**: Each connection involves: (1) `asyncio.open_unix_connection()`, (2) reading the `init_status` JSON line, (3) the actual RPC, (4) `writer.close()` + `wait_closed()`. Two full handshakes per prompt doubles the socket overhead.

**Recommendation**: Either reuse the first connection for the query, or maintain a backend-side model-to-provider mapping to eliminate the resolve round-trip entirely.

### P3 -- MEDIUM: Unbounded `_locks` dictionary growth

**File**: `container.py:41`

**What**: `self._locks` grows without bound. Every unique `user_id` adds an entry, and entries are never removed -- not even when containers are destroyed.

**How it works**: `_get_lock()` checks if a lock exists for the user, creates one if not, and returns it. `destroy_container()` removes the container from `_containers` but leaves the lock in `_locks`.

**Why this matters**: Each `asyncio.Lock` is small (~few hundred bytes), but with 100K unique users over weeks of operation, this becomes tens of MB of unreclaimable memory.

**Recommendation**: Remove the lock entry in `destroy_container()`:
```python
self._locks.pop(user_id, None)
```

### P4 -- MEDIUM: O(n^2) write amplification in batched persistence

**File**: `stream.py:294-309`

**What**: Every `_PERSIST_BATCH_SIZE` chunks, the full `accumulated` string is written to SQLite via `UPDATE messages SET content = ?`. As the response grows, each batch writes progressively more data.

**How it works**: At chunk 10, the UPDATE writes ~1KB. At chunk 100, it writes ~10KB. At chunk 500, it writes ~50KB. The total bytes written across all flushes is O(n^2) where n is the total response size.

**Why this matters**: For typical responses (5-20KB), this is fine. For long agent outputs (100KB+), the cumulative write I/O is ~50x the final response size.

**Recommendation**: Consider writing only on stream completion (single final write), or appending deltas rather than rewriting the full text each time.

### P5 -- MEDIUM: No maximum container count

**What**: There is no global limit on concurrent containers. With 1000 simultaneous users, 1000 containers at 512MB each could consume 512GB of RAM.

**Recommendation**: Add a `container_max_count` setting and enforce it in `ensure_container()`. Return 503 when the limit is reached.

### P6 -- LOW: Floating-point drift in socket polling

**File**: `container.py:150-161`

**What**: `elapsed += self._socket_poll_interval_s` accumulates floating-point error and does not account for actual `asyncio.sleep` duration.

**Recommendation**: Use `time.monotonic()`:
```python
deadline = time.monotonic() + self._socket_poll_timeout_s
while time.monotonic() < deadline:
    if os.path.exists(socket_path):
        return
    await asyncio.sleep(self._socket_poll_interval_s)
```

### P7 -- LOW: Synchronous `_ensure_network` in `__init__` blocks at startup

**File**: `container.py:45`

**What**: `_ensure_network()` makes synchronous Docker API calls during `__init__`, which runs inside the async `lifespan` function.

**Recommendation**: Move to a lazy `async def initialize()` method or wrap in `asyncio.to_thread`.

## Code Quality Findings

These findings come from the kieran-python-reviewer agent's analysis of Python conventions, type safety, and best practices.

### Q1 -- HIGH: Missing type annotations on multiple functions

**Files**: `container.py:37`, `stream.py:103,136`, `repos.py:41`

**What**: Several functions lack type hints on parameters:

```python
# container.py:37 -- missing types for both parameters
def __init__(self, settings, docker_client=None) -> None:

# stream.py:103 -- tenant parameter untyped
def _validate_workspace_path(tenant, workspace_path: str) -> None:

def _lookup_session(db, session_id: str, request: Request):

# repos.py:41 -- row parameter untyped
def _check_repo_owner(row, request: Request) -> None:
```

**Why this matters**: The project's PYTHON.md mandates type hints on all function signatures. Missing hints make the code harder to reason about and prevent `mypy` from catching type errors.

**Recommendation**: Add explicit types:
```python
def __init__(self, settings: Settings, docker_client: docker.DockerClient | None = None) -> None:
def _validate_workspace_path(tenant: TenantContext, workspace_path: str) -> None:
def _lookup_session(db: sqlite3.Connection, session_id: str, request: Request) -> sqlite3.Row | None:
```

### Q2 -- HIGH: `datetime.utcnow()` deprecated in Python 3.12+

**Files**: `container.py:26-27,71,167,182`, `test_container.py` (6+ callsites)

**What**: `datetime.utcnow()` was deprecated in Python 3.12. It returns a naive datetime with no timezone info.

**Recommendation**: Replace with `datetime.now(datetime.UTC)` or `datetime.now(timezone.utc)`.

### Q3 -- MEDIUM: Broad `except Exception` in `sidecar.py:63-66`

**What**: `SidecarClient.disconnect()` catches all exceptions silently:
```python

    await self._writer.wait_closed()
except Exception:
    pass
```

**Why this matters**: This swallows unexpected errors (e.g., `RuntimeError` from event loop issues). Expected failures are `OSError` subclasses.

**Recommendation**: Catch `OSError` specifically, log others at debug level.

### Q4 -- MEDIUM: Missing exception chaining in HTTP error handlers

**File**: `stream.py:207,225`

**What**: Exception information is lost when re-raising as HTTPException:
```python
except (ContainerStartError, ContainerNotReadyError) as exc:
    raise HTTPException(status_code=503, detail=str(exc))  # Missing: from exc
```

**Recommendation**: Add `from exc` to preserve the traceback for debugging.

### Q5 -- MEDIUM: `query` return type should be `AsyncGenerator`, not `AsyncIterator`

**File**: `sidecar.py:119`

**What**: The method uses `yield`, making it a generator, but is typed as `AsyncIterator[dict]`.

**Recommendation**: Use `AsyncGenerator[dict[str, Any], None]`.

### Q6 -- MEDIUM: Module-level mutable state for active sessions

**File**: `stream.py:36-37`

**What**: `_active_sessions` and `_active_sessions_lock` are module-level globals. The `asyncio.Lock()` created at import time could be problematic if the module is imported before the event loop starts.

**Recommendation**: Move to `app.state` alongside `container_manager`.

### Q7 -- LOW: Unused imports in test_container.py

**File**: `test_container.py:6`

**What**: `AsyncMock` and `patch` are imported but never used.

### Q8 -- LOW: `_build_options` returns untyped `dict`

**File**: `sidecar.py:88-96`

**Recommendation**: Use `dict[str, str | None]` or define a `TypedDict`.

## Simplification Findings

These findings come from the code-simplifier agent's analysis of redundancy and unnecessary complexity.

### C1 -- Dead instance fields in sidecar.js

**File**: `sidecar.js:64-65,77-78`

**What**: `this.authStorage` and `this.modelRegistry` are set in `initialize()` but never read after the per-session auth isolation change. Each session creates its own `sessionAuth` and `sessionRegistry` in `_createPiSession()`.

**Why**: These fields were needed before per-session isolation was added. Now they are dead code that confuses readers about where auth state lives.

### C2 -- Subscription leak on model change in sidecar.js

**File**: `sidecar.js:266-273`

**What**: In `processQuery`, when the model key changes, the old `entry.piSession` is disposed, but `entry.unsubscribe` from the previous subscription is not called before disposal.

**Why**: If `dispose()` does not automatically unsubscribe, this is a resource leak. The unsubscribe should be called before disposal.

### C3 -- `initialize()` return value never used in sidecar.js

**File**: `sidecar.js:80`, called from `index.js`

**What**: `initialize()` returns `{ success: true }` but the caller discards it.

### C4 -- `_loadDotEnv` does not strip surrounding quotes

**File**: `sidecar.js:96`

**What**: `.env` values with `KEY="value"` will set the value to `"value"` (with quotes). Standard dotenv libraries strip them.

### C5 -- `_lookup_session` returns different column sets depending on tenant mode

**File**: `stream.py:136-158`

**What**: The tenant-mode query omits `owner_email`, while the non-tenant query includes it. A single query with a LEFT JOIN that always includes the column would be simpler and less fragile.

### C6 -- Redundant `getattr` in main.py shutdown

**What**: `getattr(app.state, "container_manager", None)` is unnecessary because line 42 already sets `app.state.container_manager = None` in the else branch. `app.state.container_manager` is always defined.

## Test Coverage Gaps

The existing tests cover happy paths well but miss these scenarios:

1. **Concurrent `ensure_container` calls** for the same user (validates per-user locking)
2. **Concurrent `ensure_container` calls** for different users (validates global lock contention)
